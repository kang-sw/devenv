# Executor Wrapup

`executor-wrapup.md` is a shared infra playbook loaded by `ws:implement` at the end of
its doc-pipeline step, and by `ws:sprint` at wrap-up. It handles three responsibilities:

- `§Doc Pipeline` — refresh `ai-docs/_index.md`.
- `§Doc Commit Gate` — auto-commit any dirty `ai-docs/` state after doc-pipeline outputs.
- `§Ticket Update` — append result entries and move tickets when ticket-driven.

## Entry Points

- `claude-plugin/infra/executor-wrapup.md` — the canonical playbook; the single source of truth for all three responsibilities above.
- `claude-plugin/skills/implement/SKILL.md` §Step 7 — shows how `ws:implement` calls executor-wrapup directly post-merge (updaters already ran in step 4).
- `claude-plugin/skills/sprint/SKILL.md` §On:wrap-up — shows how `ws:sprint` defers the doc pipeline to a single session-end wrap-up and calls executor-wrapup after spec/mental-model updaters complete.

## Module Contracts

- `executor-wrapup` guarantees: the doc-commit gate always runs, even when no doc changes are expected. Prior steps may have dirtied `ai-docs/` without the executor knowing.
- `executor-wrapup` guarantees: it does **not** dispatch `ws:update-spec` or `ws:mental-model-updater`. Updater dispatch is the caller's responsibility, following the pattern for that executor series.
- The doc-commit gate uses `git status --porcelain ai-docs/` as the trigger. A non-empty result mandates a commit; an empty result is a no-op.
- `executor-wrapup §Ancestor Loading` invariant: when a skill loads a sub-domain mental-model doc (`mental-model/<domain>/<sub>.md`), it must also load `mental-model/<domain>/index.md` first. `ws-list-mental-model` auto-emits the parent `index.md` alongside any direct-child sub-domain — callers using it need no manual action. Callers using manual paths must read the parent before the child. Subagent spawn prompts must include the ancestor loading rule verbatim so the subagent observes it inside its own read ordering.
- `ws:write-code` file-based review protocol: reviewers write full findings to pre-allocated file paths; each reviewer returns only a `[clean|non-clean]: <one-line summary>` to the lead. The lead reads summaries only. When non-clean, the lead relays the file paths (not content) to the implementer. The implementer reads files directly. On re-review, reviewers overwrite the same paths. `ws:implement` delegates this protocol to `ws:write-code` — it does not manage reviewers directly.
- `ws-review-path` guarantees: paths are non-reproducible after the call returns (they embed a per-call `run_id`). The lead must invoke `ws-review-path <stem1> <stem2> ...` once in a single Bash call and capture all output lines. Re-invoking `ws-review-path` with the same stems yields different paths.
- `ws-review-path` cleanup: `ws:write-code` step 7 (Cleanup) issues `rm -f <correctness-path> <fit-path> <test-path>` using the literal paths stored from the single allocation call. `ws:edit` step 5 (Cleanup) issues `rm -f <review-path>`. Cleanup is mandatory; paths are in `/tmp/claude-reviews/`.
- `ws-print-infra <name>` prints an infra doc to stdout. Accepts either a bare stem (e.g., `executor-wrapup`) or a full filename with the `.md` extension. When the argument has no `.md` suffix and the exact name is not found, it probes `<name>.md` automatically. Exits 1 if neither probe succeeds.
- `ws-infra-path` portability: with `ws-new-named-agent`, pass the bare stem via `-p <stem>` — `ws-named-agent` strips a leading `ws:` prefix and resolves the remainder against `infra/prompts/`, then `infra/`, then cwd automatically. Never use bare `claude-plugin/infra/<doc>` literals (relative path, fails outside the plugin repo). Use `ws-infra-path` only when an absolute path string is needed outside `ws-new-named-agent` (e.g., `cat "$(ws-infra-path impl-playbook.md)"`). `ws-infra-path` exits 1 if the named doc does not exist under `claude-plugin/infra/`.
- `ws-named-agent new` doc-system injection: when registering a new agent, `ws-named-agent new` prepends `claude-plugin/infra/workflow-for-agent.md` (falling back to `doc-system.md` for pre-refactor installs) to the stored system prompt. Stored form: `[workflow-for-agent content]\n\n---\n\n[caller system prompt]`. Pass `--no-doc-system` to suppress injection for narrow-role agents that do not need orientation (e.g., sprint-survey, project-survey, compression helpers). If neither file exists (non-ws projects), the flag is silently ignored.
- `ws-new-named-agent` / `ws-call-named-agent` named registry: agent slots must be registered via `ws-new-named-agent <name> -p <prompt-stem>` (or legacy `--system-prompt`) before any `ws-call-named-agent` call for that name. With `-p`, prompt document bodies are stripped of frontmatter and concatenated; the first frontmatter `model:` value sets the registry tier unless `--model` overrides it. `ws-new-named-agent` stores model, tier, backend, agent type, and system prompt in `$(git rev-parse --git-dir)/ws@<repo-dir>/agents/<name>.json` and issues a fresh session UUID. `--model` accepts either an abstract tier (`opus`, `sonnet`, `haiku`) or a specific model string; backend shorthand names (`claude`, `codex`, `gemini`) are also accepted as `--model` values to select a backend without a specific model. The path uses `git rev-parse --git-dir` (not `$REPO_ROOT/.git`) so the registry is worktree-safe. `ws-call-named-agent <name> <prompt>` reads all config from the registry — model, system prompt, session routing — with no per-call flags. Before invoking the backend, it resolves the effective model via: `WS_OVERRIDE_<TIER>` env var > `~/.claude/kang-sw-devenv/ws/<escaped-repo-path>.json` `agent-overrides.<tier>` config key (falling back through global `~/.claude/kang-sw-devenv-ws.json`) > stored model. Override resolution applies only when the registry contains a `tier` field; a specific model string stored without a tier is not overridable via this mechanism. Calling `ws-call-named-agent` before `ws-new-named-agent` exits 1 with "agent not found". Output is written to both stdout and `<registry-dir>/<name>.output.txt`; the file is overwritten on every call.
- `ws-print-named-agent-output <name>` prints the last response written by the named agent. Use this after a background `ws-call-named-agent` completes to read its output. Exits 1 if no output file exists for the agent.
- `ws-named-agent erase <name>` deletes the registry JSON, outbox, output file, and the associated backend session file when the stored UUID can be resolved. `ws-oneshot-agent` depends on erase in an EXIT trap, so changing erase semantics can leave ephemeral agents or session JSONL files behind.
- `ws-oneshot-agent` wraps `new` → `call` → `erase` for one-shot, full-tool agents. It requires at least one `-p` prompt stem, reads prompt text from stdin when no positional prompt is provided, and is the execution layer behind `ws-subquery`; failures still run the cleanup trap.
- `ws-subquery` is no longer a direct restricted `claude -p` wrapper. It delegates to `ws-oneshot-agent -p subquery` with `--model haiku` or `--model sonnet` for `--deep-research`, so tool access is governed by the `subquery` prompt rather than CLI `--allowed-tools` restrictions. The subquery prompt constrains tool access to Explore level: Read, Bash (read-only commands), Glob, Grep, WebFetch, WebSearch — Edit, Write, NotebookEdit, and Agent are excluded. ws-subquery accepts a question as a positional argument, as piped stdin, or as a here-doc via the `-` argument (`ws-subquery - <<'PROMPT'`).
- `ws-named-agent tail <name> [-<n>]` reads the last N assistant turns from the live session JSONL on disk without invoking the CLI — safe to call while the agent is running. Defaults to 3 turns. Backend-aware: claude sessions are read from `~/.claude/projects/`; codex sessions from `~/.codex/sessions/`. Use the `[last]` line to distinguish agent state: a tool-results line means the agent is awaiting the next turn; an assistant turn with tools means it is waiting on tool results; an assistant turn with no tools means it has concluded.
- `ws-named-agent override <tier> <model>` persists a model override for a tier to the local config under `~/.claude/kang-sw-devenv/ws/`; pass `-g` to write the global `~/.claude/kang-sw-devenv-ws.json` instead. `ws-named-agent override <tier> --unset` removes it. `ws-named-agent override show` prints the current config. Override takes effect on the next `ws-call-named-agent` call without re-registering the agent. Valid tiers: `opus`, `sonnet`, `haiku`.
- `ws-interrupt-named-agent <name> <message>` queues a message for the named agent as a new user turn. The message is appended to `<registry-dir>/<name>.outbox.txt`. If the agent is currently running, the `PostToolBatch` hook (`ws-agent-check-mailbox`) exits 2 at the next tool boundary; `ws-call-named-agent`'s drain loop then resumes the session with the queued content. If the agent is idle, the message is delivered on the next `ws-call-named-agent` call. Multiple `ws-interrupt-named-agent` calls before the drain loop fires append without a separator — they arrive as one concatenated user turn. The `--settings` flag on the underlying `claude` invocation accepts a raw JSON string (not only a file path); this is how the `PostToolBatch` hook is injected inline without a temp file.
- `ws:write-code` background-mode default: all `ws-call-named-agent` Bash calls in the write-code skill (implementer spawn, reviewer spawns, re-review loop) use `run_in_background: true`. The lead reads each agent's output via `ws-print-named-agent-output <name>` after the completion notification arrives — not from stdout.
- `ws-call-named-agent` retry: when the `claude` subprocess exits non-zero and stderr contains "already in use", `ws-call-named-agent` retries up to 3 times with a 3-second delay before propagating the error. The retry loop is transparent to callers; no special handling is required. If all retries fail for any other reason, the script exits 1 immediately.
- `ws-call-named-agent` auto-compression: applies to the `claude` backend only. When `token_count > 120K` tokens, `ws-call-named-agent` transparently replaces the active session with a fresh one. The 3-step flow is: extract intent via haiku, compress session state via the model's own context, hand off to a new agent session with compressed context prepended. The lead observes no behavioral difference — the handoff result appears on stdout as normal. Re-compression is suppressed on the immediate next call via a `compressed_at` flag in the registry. The `codex` backend has compression disabled; token count is tracked for observability but no handoff occurs. Token count is backend-dependent: the claude backend sums `input_tokens + cache_creation_input_tokens + output_tokens`; the codex backend sums `input_tokens + cached_input_tokens + output_tokens`. The registry records the computed value in both `token_count` and `context-window`.
- `implementer.md` review findings guidance: when review findings arrive as a file path, the implementer reads the file then applies judgment — address correctness, contract, and security findings; deprioritize style or naming feedback that conflicts with established codebase patterns; never apply a finding without understanding why it matters for the specific change.
- `ws:sprint` task routing: single-file isolated changes route to `ws:edit` (one reviewer covering correctness+fit; 2-cycle cap; `ws-review-path direct`). Multi-file or new-pattern changes route to `ws:write-code` (three reviewers; 3-cycle cap; `ws-review-path correctness fit test`). Sprint does not manage reviewer agents directly — both primitives self-register and self-clean their review path files.
- `ws:sprint` doc-pipeline deferral: doc pipeline does not run after individual tasks. It runs once at `On:wrap-up` in this order: (1) spec-update pass — invoke `ws:update-spec` with the sprint commit range (`$PARENT..HEAD`); commit immediately, (2) `ws:mental-model-updater` (after the spec-update pass so it sees the updated spec); commit immediately, (3) executor-wrapup. Each update is committed in its own commit right after it completes — do not defer or batch both into a single commit. After executor-wrapup, sprint auto-merges via `ws-merge-branch main sprint/<name>` — the merge is not manually suggested.

## Coupling

- `ws:edit` → caller (`ws:implement`): edit invokes `ws:update-spec` on its commit range before returning; it outputs commit range, test status, and spec result to the caller. `ws:implement` is responsible for dispatching `ws:mental-model-updater` after edit returns.
- `ws:implement` → `ws:update-spec` + `ws:mental-model-updater` → `executor-wrapup`: implement invokes ws:update-spec then dispatches mental-model-updater sequentially in its doc pre-pass (step 4, before merge). After merge it calls executor-wrapup. The commit gate captures any post-merge doc changes.
- `ws:write-code` → caller (`ws:implement`): write-code is a primitive with no doc pipeline. It self-cleans review path files and outputs a commit range, brief path, and test status to the caller.
- `ws:sprint` → `ws:update-spec` → `ws:mental-model-updater` → `executor-wrapup` → `ws-merge-branch`: the session suppresses per-task doc pipeline. At wrap-up, sprint invokes `ws:update-spec` with the sprint commit range, commits spec changes, then dispatches `ws:mental-model-updater`, commits that output, then calls executor-wrapup, then auto-merges via `ws-merge-branch main sprint/<name>`.

## Extension Points & Change Recipes

- **Add a new executor skill**: decide which pattern to follow:
  - Primitive (no doc pipeline, no merge): return commit range + test status to caller. Self-clean review paths. Do not dispatch updaters.
  - Harness-like (orchestrates a primitive, owns doc pipeline, owns merge): invoke the primitive, invoke ws:update-spec then dispatch mental-model-updater sequentially in a doc pre-pass, merge, then call executor-wrapup.
  - Sprint-like (session container, feature branch): suppress per-task doc pipeline; at session end invoke `ws:update-spec` with the session commit range, commit spec changes, then run `ws:mental-model-updater`, then call executor-wrapup, then auto-merge.
  Mixing the patterns — calling executor-wrapup before updaters finish, or dispatching updaters inside executor-wrapup — breaks the commit-gate capture guarantee.
  Include the ancestor loading rule in any implementer spawn prompt when that implementer may read sub-domain mental-model docs. Use `ws-list-mental-model` rather than manual paths where possible — it handles ancestor emission automatically.
- **Change wrapup responsibilities**: edit `claude-plugin/infra/executor-wrapup.md` only. Do not duplicate wrapup logic in individual skill files.

## Common Mistakes

- Dispatching `ws:update-spec` or `ws:mental-model-updater` inside executor-wrapup — the playbook intentionally excludes updater dispatch. Adding it there causes double-dispatch for `ws:implement`, whose updaters already ran pre-merge.
- Calling executor-wrapup before updaters finish in a harness skill — the commit gate fires before updater outputs exist, leaving ai-docs/ changes uncommitted.
- Assuming the commit gate is a no-op when no doc changes were planned — prior steps such as updater agents always produce at least a checkpoint commit that touches `ai-docs/`.
- Calling `ws-review-path` twice expecting the same paths — paths embed a per-call `run_id` and are non-reproducible. Allocate once; store the paths; use them throughout the review loop.
- Using bare `claude-plugin/infra/<doc>` as a `-p` or `--system-prompt` value — this path is relative to the plugin repo and fails in downstream projects. Use `-p <stem>` instead; `ws-new-named-agent` strips an optional `ws:` prefix and resolves the stem against `infra/prompts/` then `infra/` automatically. Use `ws-infra-path` only when an absolute path is needed outside `ws-new-named-agent`.
- Relaying reviewer file contents to the implementer instead of file paths — the protocol sends paths. The implementer reads files directly. Sending content bypasses the file-based protocol and inflates lead context unnecessarily.
- Omitting the `rm -f` cleanup at step 8 — review files persist in `/tmp/claude-reviews/` across runs if not deleted, potentially leaking findings from a prior review cycle.
- Calling `ws-interrupt-named-agent` multiple times before the drain loop cycles expecting separate user turns — the outbox is a flat file; multiple appends concatenate without a separator, and the drain loop delivers all pending content as one user turn.
- Running doc pipeline per task inside a `ws:sprint` session — the session's invariant defers all doc pipeline to wrap-up. Running the spec-update pass or dispatching `ws:mental-model-updater` mid-session creates partial checkpoints that confuse the wrap-up updater run.
- Batching the spec-update pass and mental-model-updater outputs into one deferred commit at wrap-up — each must be committed immediately after it completes. Batching obscures which step produced which change and makes the `(mental-model-updated)` checkpoint stamp appear on a mixed commit.
- Dispatching `ws:mental-model-updater` inside `ws:edit` — edit invokes `ws:update-spec` internally but has no mental-model pipeline. Dispatching `ws:mental-model-updater` inside edit causes double-dispatch when `ws:implement` calls edit then runs its own doc pre-pass.
