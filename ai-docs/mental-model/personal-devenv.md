# Personal Devenv

## Entry Points

- `install.sh` — bootstrap/update script; owns symlinks, Claude Code settings, plugin snapshot sync, marketplace registration, and plugin install/update checks.
- `.claude-plugin/marketplace.json` — repo-local marketplace metadata for directory-based plugin discovery from the live repository.
- `claude-plugin/.claude-plugin/plugin.json` — ws plugin manifest copied into the snapshot cache.

## Module Contracts

- `install.sh update` copies `claude-plugin/` into `~/.claude/plugins/ws-plugin/ws/` with `rsync --delete`, then generates `~/.claude/plugins/ws-plugin/.claude-plugin/marketplace.json` whose plugin `source` is `./ws`. Claude Code is registered against the snapshot marketplace root, not the live repository.
- Plugin install idempotence is path-sensitive: `install.sh` treats `ws@kang-sw-devenv` as current only when `installed_plugins.json` records `installLocation == ~/.claude/plugins/ws-plugin`. If the plugin is installed at an older path, the script deletes the stale installed-plugin entry before running `claude plugin install`.
- `settings.json` and `known_marketplaces.json` must both point the `kang-sw-devenv` marketplace at the snapshot cache. Pointing only one of them at the cache can leave `/reload-plugins` and fresh `claude plugin install` resolving different plugin trees.
- The global Claude Code instruction file is a symlink from `~/.claude/CLAUDE.md` to `claude-plugin/CLAUDE.home.md`. Edits to the home-level doctrine belong in the repo copy, then require rerunning `install.sh update` only for plugin snapshot changes, not for the symlinked `CLAUDE.md` itself.

## Coupling

- `install.sh` ↔ `claude-plugin/`: install copies the plugin directory into the snapshot cache. Live edits under `claude-plugin/` do not affect the running plugin until the snapshot is refreshed.
- `install.sh` ↔ `.claude-plugin/marketplace.json`: the repo-level marketplace still points at `./claude-plugin`, but the installed machine-local marketplace generated by install points at `./ws` under the snapshot cache. These two marketplace files intentionally have different source paths.
- Plugin cache ↔ `claude-plugin/bin/*.cmd`: every new bin script needs a Windows-compatible shim before snapshot sync; otherwise the cached plugin is only usable from Unix-like shells.

## Extension Points & Change Recipes

- **Change plugin cache layout**: update all three places together — the `PLUGIN_CACHE`/`rsync` destination in `install.sh`, the generated snapshot marketplace `source`, and the `installed_plugins.json` installLocation check. Leaving one stale causes version no-op installs or undiscoverable plugins.
- **Add a new plugin binary**: add the script under `claude-plugin/bin/`, add the matching `.cmd` shim, verify both shells, then run `install.sh update` so the cache receives both files.

## Common Mistakes

- Editing `claude-plugin/` and expecting Claude Code to pick it up immediately — the active plugin reads from `~/.claude/plugins/ws-plugin/ws/` after install. Refresh the snapshot.
- Reusing the repo-level marketplace `source: ./claude-plugin` inside the snapshot marketplace — from the snapshot root the plugin directory is `./ws`, so discovery fails.
- Removing the stale-entry purge from `install.sh` — Claude Code may no-op an install when the version matches, preserving an old `installLocation` even though the marketplace path changed.
