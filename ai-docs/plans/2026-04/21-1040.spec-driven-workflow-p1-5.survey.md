# Survey: Spec-Driven Workflow (Phases 1–5)

## Reusable Components

- `claude/bin/spec-build-index` — `update_frontmatter(fm, features_text)`: replaces or
  inserts a named YAML block in frontmatter by scanning for the key line and
  replacing its indented continuation. The `stems:` block update (Phase 1) is
  a direct analog — write `update_stems(fm, stems_text)` using the same
  find-and-replace loop pattern.

- `claude/bin/spec-build-index` — `extract_features(body)`: returns `(depth, title)`
  pairs from `##`+ headings via `HEADING_RE = re.compile(r"^(#{2,6})\s+(.*)")`.
  Phase 1 reuses this regex but must also extract the `{#slug}` anchor from the raw
  title string (`re.search(r'\{#([\w-]+)\}', title)`). Headings without a match are
  skipped for stems; all headings feed `features:` as before.

- `claude/bin/list-mental-model` — `parse_frontmatter(path)`: a full YAML frontmatter
  parser (scalars, sequences, nested maps) at lines 26–77. `list-stems` needs to
  read the `stems:` map from spec frontmatter; this parser handles that format and
  can be copied or imported directly.

## Existing Patterns

- **bin command style** — two patterns coexist: Python scripts with `#!/usr/bin/env
  python3` and a module-level docstring (`spec-build-index`, `list-mental-model`);
  bash scripts with `#!/usr/bin/env bash` and `set -euo pipefail` (`load-infra`,
  `merge-branch`). `list-stems` operates on frontmatter YAML, so the Python pattern
  fits. See `claude/bin/list-mental-model` for the full Python template including
  `sys.stdout.reconfigure(encoding='utf-8')` and `main()` guard.

- **Frontmatter block ownership** — `spec-build-index` owns the `features:` block;
  it preserves all other frontmatter fields by only replacing the one block it owns.
  `stems:` should follow the same ownership model: a distinct block `spec-build-index`
  regenerates, leaving `title`, `summary`, and `features:` untouched.

- **SKILL.md judge block** — judges are defined under `## Judgments` as
  `### judge: <name>` sections with a decision rule body. See `write-spec/SKILL.md`
  lines 39–49 (`judge: directory-vs-flat`, `judge: split-trigger`). The new
  `judge: spec-impact` (Phase 2) adds a third entry to that section.

- **On-invoke step numbering** — steps in `## On: invoke` are numbered starting at 1.
  Phase 2 inserts a new step 0 before the existing numbered steps; the existing steps
  renumber to 1–5 (currently 1–5, becomes 2–6 if 0 is used, or 1 becomes the guard
  and original 1 becomes 2 — choose one convention and apply consistently).

- **Skill chaining suggestion** — `write-ticket/SKILL.md` Step 6 "Spec check" prompts
  the user to invoke `/write-spec` when public behavior changes. Phase 3 moves this
  prompt to `/discuss`'s on-user-signals-done and makes it unconditional. The
  existing Step 6 wording is the model: adapt it to use `list-stems` for stem lookup.

## Constraints

- **`features:` includes raw heading text including `{#slug}`**: `extract_features`
  currently takes `m.group(2).strip()`, which includes the anchor syntax in the
  output. After Phase 1, spec files will have headings like
  `## Feature Name {#feature-slug}`, and the auto-generated `features:` block will
  contain those strings verbatim. The implementer must decide: strip anchors from
  features display text too, or leave them. Stripping is cleaner but is a behavior
  change to `spec-build-index` beyond the ticket scope.

- **Stem hierarchy requires parent-slug tracking**: `extract_features` returns flat
  `(depth, title)` pairs with no parent linkage. The stem generator needs to maintain
  a `parent_slugs[depth]` dict updated as it scans headings — when a shallower
  heading is encountered, deeper entries are cleared. This is not a reuse of existing
  logic; it is new.

- **`spec-build-index` must handle the `stems:` field as a YAML map, not a list**:
  the existing `update_frontmatter` indents each line of a pre-built string. For
  `stems:`, the value per key is a quoted string (`"## Heading Text"`). The builder
  must emit `  key: "value"` lines, not `  - item` lines. The `_indent_block`
  helper does not distinguish — the caller must pre-format the stems block string
  differently from the features block string.

- **`list-stems` file path argument is relative to the caller, not to `ai-docs/spec/`**:
  the stem path component (`<file-stem-path>`) is relative to `ai-docs/spec/`. The
  command must compute this relative path from the given file argument regardless of
  the caller's working directory. See how `spec-build-index` uses `Path(arg)` with
  no base — `list-stems` needs to resolve the spec-relative path explicitly
  (e.g., `path.relative_to(Path('ai-docs/spec'))` after resolving to absolute).

- **CLAUDE.md commit convention** (Phase 4): the commit template is in
  `/home/swkang/devenv/claude/CLAUDE.md` (symlinked to `~/.claude/CLAUDE.md`),
  not in the project `CLAUDE.md`. Edits land in `claude/CLAUDE.md` and are surfaced
  by `git diff`. The `## Spec` section addition goes inside the existing commit
  message template block.
