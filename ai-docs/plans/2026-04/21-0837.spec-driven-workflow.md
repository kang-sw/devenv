# Spec-Driven Workflow — Spec-Stem System and Mandatory Gate

## Context

Single coordinated sweep covering all six phases of `260420-feat-spec-driven-workflow`.
The ticket's own decisions stand; this plan only resolves the implementation
contracts and sequencing the ticket leaves open.

Sequencing is locked by dependency, not by ticket phase number:

- **Block A (tooling)** — `ws-spec-build-index` extension, new `list-stems` bin.
  Prerequisite for every downstream edit that calls either tool.
- **Block B (skill + agent edits)** — `write-spec` judge, `write-spec` format
  change, `/discuss` unconditional suggestion, `write-ticket` prompt,
  `spec-updater` protocol, convention docs (`ticket-conventions.md`,
  `claude/CLAUDE.md`). Block A must be in place so any example or
  self-test referencing `list-stems` resolves.
- **Block C (migration + canonical flow)** — `_index.md` flow, enter-session
  Workflow Map, existing spec file scrub. Runs last so the migrated spec files
  match the new conventions Block B enforced.

Key ticket decisions that bind implementation choices:

- Stems are **auto-derived**, never hand-authored. The executor must not add
  a stem-authoring affordance anywhere.
- `🚧` on spec headings carries **no** `[ticket-stem/pN]` suffix. Ticket →
  spec reference flows only one way: tickets name spec-stems, specs never
  name tickets.
- The `spec-impact` judge lives inside `/write-spec`, not `/discuss`. Discuss
  always suggests write-spec; write-spec's step 0 decides no-op vs. proceed.

Data contracts crossing capsule boundaries — flagged for review if any drift
from the ticket:

- Stem format string grammar (consumed by `list-stems`, `write-ticket`,
  commit-message convention, `spec-updater` grep).
- `stems:` YAML frontmatter shape in spec files (consumed by `list-stems`).
- `spec:` optional frontmatter field in ticket files (consumed by humans
  and future tooling).
- `## Spec` commit trailer block and `renamed-spec:` line (consumed by
  `spec-updater`'s `git log --grep`).

## Steps

### Block A — Tooling

#### A1. Extend `ws-spec-build-index` with `stems:` generation

File: `claude/bin/ws-spec-build-index`.

Add stem generation alongside the existing features-tree generation. Both
fields must survive re-runs — the current `update_frontmatter` helper
replaces a single named block by prefix-scan; the same helper pattern
(`find start by line.startswith("<field>:")`, replace block) applies to
`stems:`. Call it once per field inside `process_file`.

Stem derivation contract (exact):

```
stem = "<file-stem-path>:<h2-slug>[.<h3-slug>[.<h4-slug>...]]"

file-stem-path:
  path of the spec file relative to ai-docs/spec/, with .md stripped,
  using "/" as directory separator. "index.md" is treated mechanically —
  ai-docs/spec/skills/index.md → "skills/index", NOT "skills".

slug normalization (per heading segment):
  1. Lowercase the heading text.
  2. Replace ASCII whitespace runs with single "-".
  3. Strip every character that is not [a-z0-9-]. This removes 🚧, (, ),
     [, ], punctuation, and any Unicode letter. Hyphens inserted in step 2
     survive.
  4. Collapse runs of "-" to a single "-".
  5. Strip leading and trailing "-".
```

All `##` and deeper headings produce a stem. The document title (`#`)
does not. Example matching the ticket:

```yaml
stems:
  skills:workflow-skills: "## Workflow Skills"
  skills:workflow-skills.discussion: "### Discussion"
  skills:workflow-skills.discussion.discussion-loop: "### Discussion Loop"
```

Output shape: `stems:` is a YAML block mapping `<stem>` → `"## Original
Heading Text"`. The value is the original heading line with leading `#`
markers and surrounding whitespace preserved (`## Workflow Skills`,
`### Discussion`). Quoting: always double-quote the value string,
escaping `"` and `\` the same way as other YAML emitters in this repo.

Depth tracking mirrors the existing features walk — maintain a stack
of `(depth, slug)` frames where `depth 0 = ##`. On encountering a
heading at depth `d`, truncate the stack to `d` frames and push the
new slug. The stem is `file-stem-path + ":" + ".".join(stack-slugs)`.

Idempotency requirement: running `ws-spec-build-index` twice in succession
on the same file must produce byte-identical output. The existing
features-block writer already meets this; the stems-block writer must
match. In particular, preserve `title:`, `summary:`, `features:`, and
any user-authored frontmatter fields across re-runs; only `stems:` and
`features:` are regenerated.

Edge cases (executor must verify):

- Heading that normalizes to empty after slug strip (e.g. `## 🚧 🚧`) —
  fall back to `h<depth>-<N>` where `N` is the 1-based ordinal of the
  heading at that depth among its siblings under the same parent.
  Record this in a short comment in the code so the fallback is
  documented.
- Duplicate slugs within the same parent — append `-<N>` where `N` is
  the 2nd, 3rd, … occurrence. First occurrence stays unsuffixed.
- File with no frontmatter — create one with only `stems:` and
  `features:` fields, same behavior the current code already has for
  missing features.

#### A2. New `list-stems` bin script

File: `claude/bin/list-stems` (new, executable).

Shape follows `ws-list-mental-model`: single Python file, shebang
`#!/usr/bin/env python3`, reuses the same hand-rolled frontmatter
parser pattern (no PyYAML dependency — none of the existing bin
scripts import it).

Contract:

```
Usage: list-stems <spec-file-path>

Reads the stems: map from the file's YAML frontmatter and prints
one line per entry:

  <stem>\t<heading-text>

where <heading-text> is the frontmatter value with surrounding
quotes stripped. Exits 0 on success, non-zero with a stderr
message on: missing file, missing frontmatter, missing stems: key.

If stems: is present but empty, prints nothing and exits 0.
```

Output uses tab separator so downstream `cut -f1` or `awk -F'\t'`
works predictably. No sorting — emit in frontmatter order.

The parser's only job is locating the `stems:` block and iterating
its nested `  <stem>: "<heading>"` lines. The `ws-list-mental-model`
frontmatter parser (`parse_frontmatter`, lines ~26–77) is a working
template; copy and trim it to the map-only shape this script needs.

#### B6. `spec-updater` protocol rewrite

File: `claude/agents/spec-updater.md`.

Replace the ticket-lookup protocol with a grep-based one:

- **Constraints** — drop the "bare 🚧 means untracked" rule. Bare 🚧 is
  now the standard format.
- **Process step 2 "Classify each 🚧 occurrence"** — collapse the three
  categories (annotated heading, annotated Planned callout, bare) into
  one: any 🚧 on a heading or callout.
- **Process step 3 "Check completion"** — replace the ticket-file
  lookup with:

  ```
  For each 🚧 heading or callout in the file:
    a. Derive the spec-stem via `list-stems <spec-file>` — match on
       heading text rather than trying to regenerate the stem in the agent.
    b. Run: git log --oneline --all --grep=<stem>
       (the stem contains ':' and '.' — safe for grep without escaping
       special regex chars because git log --grep uses fixed-string
       matching only with -F; use plain --grep and accept the regex
       interpretation since the slug alphabet is [a-z0-9-:.]).
    c. Each match is a commit that referenced this stem.
       Classify:
       - Matches exist → prompt caller to strip 🚧 (do not strip
         speculatively; require confirmation).
       - No matches → report "not yet implemented" — no action.
  ```

- **Process step 4 "Apply confirmed strips"** — strip only 🚧 prefix,
  no more `[stem/pN]` trailer to remove. For Planned callouts, the
  removal logic simplifies to: delete the entire `> [!note] Planned
  🚧` block. Run `ws-spec-build-index` after modification as before.

- **Output report** — drop the "Untracked 🚧" section. Retain
  "Stripped", "Needs confirmation" (renamed to "Implementation
  suspected — commits found, awaiting confirmation"), and "Missing
  implementation" (replaces "Missing tickets", meaning the stem has
  zero matching commits).

- **Frontmatter `description:`** — update to match the new protocol.
  The word "ticket" no longer belongs in the description.

## Testing

Post-impl verification covers the whole sweep. No TDD, no skeleton tests.

Manual verification steps after implementation lands:

- **A1 idempotency** — pick a spec file, run `ws-spec-build-index` twice;
  `git diff` must show zero change between the two runs.
- **A1 fallback** — add a temp heading like `## 🚧 🚧` and confirm the
  emitted stem uses the `h<depth>-<N>` fallback.
- **A2 round-trip** — run `list-stems ai-docs/spec/skills.md` after A1
  has populated the field; confirm output matches the frontmatter and
  the stems reproduce the file's heading hierarchy.
- **B1 judge no-op** — invoke `/write-spec` on a clearly internal
  refactor topic; confirm the exit message names `/write-ticket`.
- **B1 judge proceed** — invoke on a public-surface change; confirm
  the flow enters the existing on-invoke steps.
- **B2 discuss chain** — simulate `/discuss` termination; confirm
  `/write-spec` appears unconditionally as the first suggestion.
- **B3 + B4 write-ticket** — author a ticket against a spec stem;
  confirm the resulting frontmatter includes `spec:` and the stem
  matches `list-stems` output.
- **B5 commit trailer** — construct a commit message with
  `## Spec` and `renamed-spec:` per the template; confirm no lint
  or hook complaints (CLAUDE.md's commit rule is prose, no enforcement).
- **B6 spec-updater** — run spec-updater against the migrated
  `skills.md`; confirm it reports "implementation suspected" for
  stems whose commits landed in this sweep, and no "Untracked 🚧"
  entries surface.
- **C1 migration sweep** — `grep -rn '\[.*-p[0-9]\+\]' ai-docs/spec/`
  returns empty.

## Success Criteria

- `ws-spec-build-index` emits both `features:` and `stems:`; both fields
  survive re-runs byte-identically.
- `list-stems` on any spec file emits tab-separated `stem\theading` lines
  matching the file's heading structure.
- `/write-spec` gates on the `spec-impact` judge; the no-op path exits
  with a suggestion to `/write-ticket` and writes nothing.
- No spec file under `ai-docs/spec/` contains `[<stem>/p<N>]`,
  `[ticket-stem/pN]`, or equivalent embedded ticket references.
- `/discuss`'s done handler unconditionally offers `/write-spec` as
  the first persistence option; no conditional "was this public?" prompt
  remains.
- `/write-ticket` prompts for spec-stem references and writes them
  into the `spec:` frontmatter field.
- `claude/infra/ticket-conventions.md` template documents `spec:`.
- `CLAUDE.md` (project root) commit rules document `## Spec` and `renamed-spec:`.
- `spec-updater` verifies 🚧 status via `git log --grep=<stem>`, does
  not require ticket links, and no longer flags bare 🚧 as an anomaly.
- `ai-docs/_index.md` canonical flow and `enter-session` Workflow Map
  both show `/write-spec` as the mandatory post-`/discuss` step.
