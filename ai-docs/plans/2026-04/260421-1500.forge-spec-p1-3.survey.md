# Survey: /forge-spec — From-Scratch Spec Reconstruction (Phases 1–3)

## Reusable Components

- `claude/skills/write-spec/SKILL.md` — `write-spec`: The most directly reusable skill. Provides the per-anchor authoring loop (`generate-spec-stem` → insert → `spec-build-index`), the `spec-format` template, and the `load-infra spec-conventions.md` runtime call. The forge-spec per-domain loop in Phase 2 replicates this pattern at scale; prefer identical step ordering to keep both skills consistent.

- `claude/bin/generate-spec-stem` — CLI: `generate-spec-stem <slug> [<slug2> ...]`. Outputs one `YYMMDD-slug` per line. Collision-safe against all existing anchors under `ai-docs/spec/`. When multiple slugs are passed in one call they are deduplicated against each other. Date prefix derived from `datetime.date.today()` — always the invocation date, not a hardcoded constant.

- `claude/bin/spec-build-index` — CLI: `spec-build-index <spec-file.md> [<spec-file2.md> ...]`. Parses `##`-and-deeper headings, rebuilds `features:` frontmatter, and silently removes any `stems:` block. Idempotent. Must be run after every spec file write. Fails with an error message to stderr and skips the file if no headings are found.

- `claude/bin/list-stems` — CLI: `list-stems [-v] [<spec-file>]`. No file arg: flat scan of all stems under `ai-docs/spec/`. With file arg: hierarchical output with heading context. `-v` adds tab-separated display label (only useful with file arg; emits warning in flat mode).

- `claude/infra/spec-conventions.md` — canonical arbiter of what goes in spec. The skill must load it at runtime via `load-infra spec-conventions.md` before any spec write, exactly as `write-spec` does. Key rules: external/behavioral only, `🚧` prefix for planned features, no `{#slug}` in frontmatter, no ticket references in `🚧` markers.

## Existing Patterns

- **Cold-start vs resume detection**: `parallel-implement/SKILL.md` and `delegate-implement/SKILL.md` both create a task list at invocation start and track it to completion. The pattern for resume detection is to check `TaskList` for existing tasks scoped to the current context on every invocation — if found, skip initialization and continue from the first incomplete task. `forge-spec` Phase 1 mirrors this exactly.

- **Subagent dispatch with explicit model override**: `write-plan/SKILL.md` shows the canonical form. Model override is always an explicit `model = "sonnet"` or `model = "opus"` field on `Agent()` — never inherited from parent. All three `Agent()` call sites in `write-plan` carry it. Forge-spec must do the same for every explore subagent.

- **Parallel subagent dispatch**: `write-plan` dispatches multiple subagents in the same message for parallel survey. `parallel-implement` spawns 2N agents in one batch. The pattern: all independent agents are spawned in the same response turn; waiting happens after all are spawned.

- **Task list as workflow state**: `parallel-implement/SKILL.md` step 9 shows the exact TaskCreate pattern — one task per scope, one task per workflow phase, all in a single registration step. `delegate-implement/SKILL.md` step 1 Prepare shows a bulleted checklist as the task list content. The task list doubles as a resumable checklist across compact boundaries.

- **User confirmation gate before destructive action**: `delegate-implement/SKILL.md` has "User approves the report before merge — no code reaches the target branch without user confirmation." Forge-spec's archive step (`git mv ai-docs/spec/*`) is the equivalent destructive action and follows the same gate pattern.

- **`spec-build-index` idempotent safety pass**: `write-spec/SKILL.md` runs `spec-build-index` both after each file write and as an explicit final pass. Phase 3 wrap-up of forge-spec mirrors this.

## Relevant Interfaces

- `claude/skills/team-lead/SKILL.md` — `TaskCreate / TaskUpdate` invocation pattern:
  - `TaskCreate` is called at preparation time to register all tasks at once.
  - `TaskUpdate` marks tasks `in_progress` or `completed` as work proceeds. The `owner` field assigns a task to a teammate by name.
  - Tasks carry their full scope description in the `description` field — this is the persistence payload that survives a compact boundary.
  - The exact field names used in `parallel-implement`: `name` (short identifier), `description` (includes file_set + test_command + brief), and status transitions via `TaskUpdate`.

- `claude/skills/write-plan/SKILL.md` — `Agent()` spawn template used for survey mode:

  ```
    model = "sonnet",
    prompt = """
      Read `${CLAUDE_SKILL_DIR}/survey-writer.md` first.

  ```
  This is the exact template shape for all forge-spec explore subagents.

- `claude/infra/spec-conventions.md` — loaded at runtime via `load-infra spec-conventions.md`. Not embedded in the skill file. The skill references it by path.

## Constraints

- **`spec-updater` does not exist** as a binary under `claude/bin/`. The ticket's Prior Art section lists it and Phase 3 suggests running it, but there is no `claude/bin/spec-updater` file. The implementer must either create it as part of this ticket or treat the Phase 3 suggestion as a forward reference to a not-yet-implemented tool.

- **`generate-spec-stem` is anchored to `ai-docs/spec/`** via `__file__`-relative path resolution (`Path(__file__).resolve().parents[2] / "ai-docs" / "spec"`). It will only find stems under that directory. During Phase 1 cold start, after `git mv ai-docs/spec/* ai-docs/ref/old-spec/YYMMDD/`, the spec directory is empty — `generate-spec-stem` will find no existing stems, so all new stems will be the base slug without a collision suffix. This is correct behavior but must not be confused with a tool failure.

- **`spec-build-index` requires at least one `##`-or-deeper heading** or it skips the file (prints "skip: no headings found in ..."). A spec file with only a `#` title and no `##` sections will not get a `features:` block. The skill must ensure all spec files have at least one `##` section before calling `spec-build-index`.

- **Anchors live in document body only, never in frontmatter.** `spec-build-index` silently removes any `stems:` block from frontmatter. The skill must not add anchor metadata to frontmatter during Phase 2 writes.

- **TaskList scope**: the ticket specifies checking TaskList for tasks "scoped to this project" for resume detection. The existing skills do not show a project-scoping filter on TaskList — the implementer must decide whether to use a naming convention (e.g., task names prefixed with `forge-spec-`) or rely on the task description to identify the forge-spec run.

- **`load-infra` is a Bash command**, not a skill invocation. `write-spec/SKILL.md` calls it as `run load-infra spec-conventions.md`. The forge-spec SKILL.md must follow this same convention when loading `spec-conventions.md`.
