# Survey: Global Unique Spec Stems ‚Äî Design Pivot (260421)

## Reusable Components

- `claude/bin/list-stems` ‚Äî `parse_stems()`: reads YAML frontmatter between `---` delimiters, locates the `stems:` key, collects indented `key: value` pairs. The frontmatter-detection and delim-search logic (lines 27‚Äì52) can be discarded entirely in the rewrite. The `main()` shell ‚Äî `sys.stdout.reconfigure`, `-v` flag parse via `sys.argv`, single-file path argument, `sys.exit` on error ‚Äî is the reuse target for `generate-spec-stem`.

- `claude/bin/spec-build-index` ‚Äî `_replace_or_insert_block()`: shared helper that locates a YAML key's block by its leading `key:` line, then replaces lines `[start:end]` in the frontmatter line list. Phase 3 reuses this to remove the `stems:` block: pass an empty `new_field_lines` list and skip the "insert if absent" branch. Currently used by both `update_frontmatter` and `update_stems`.

- `claude/bin/spec-build-index` ‚Äî `parse_frontmatter_and_body()`: splits raw file text at the `---` delimiters into `(frontmatter_str, body_str)`. The new `list-stems` rewrite can import or duplicate this pattern for splitting the file before grep-scanning the body.

- `claude/bin/list-mental-model` ‚Äî command scaffold: `sys.stdout.reconfigure(encoding='utf-8')`, `targets = sys.argv[1:]`, `print(..., file=sys.stderr)` + `sys.exit(1)` on error, `if __name__ == '__main__': main()` guard. This is the exact pattern `generate-spec-stem` must follow. `list-mental-model` additionally shows multi-file glob iteration (`sorted(dir.glob('*.md'))`) ‚Äî reuse in the no-arg path of the new `list-stems`.

## Existing Patterns

- Frontmatter key ownership: `spec-build-index` treats `features:` and `stems:` as fully owned ‚Äî it regenerates them unconditionally on each run via `_replace_or_insert_block`. Phase 3 applies the same "owned key" model to `stems:`: the block is replaced with nothing (removed), not left stale.

- Heading regex: `HEADING_RE = re.compile(r"^(#{2,6})\s+(.*)")` in `spec-build-index` (line 16). The new `list-stems` uses the same pattern to detect heading lines and extract depth and raw title.

- Anchor regex: `SLUG_RE = re.compile(r"\{#([\w-]+)\}")` in `spec-build-index` (line 18). The new `list-stems` uses the same pattern to extract stem keys from heading and body lines. For the new globally-unique format, the pattern from the lead directives is `\{#\d{6}-` (requires 6-digit date prefix).

- Stderr / exit on empty result: `list-stems` currently prints a hint to stderr and exits 0 when no stems are found (lines 72‚Äì75). Same pattern applies to the new implementation for the no-results case.

- `-v` verbose flag: `list-stems` implements it via `if args and args[0] == "-v": verbose = True; args = args[1:]` (lines 59‚Äì62). The new `list-stems` reuses this exact pattern.

## Relevant Interfaces

- `claude/bin/list-stems` ‚Äî current output format: flat `key` or tab-separated `key\tdisplay` per line. The new format changes to indented hierarchical output: heading-level stems at column 0, body-level stems indented one level under their parent heading. The `-v` flag adds the heading/nearby text label after the stem key (two-space-separated columns per ticket example).

- `claude/bin/spec-build-index` ‚Äî `process_file()` (lines 154‚Äì171): the integration point where `extract_stems` and `update_stems` are called. Phase 3 removes lines 166‚Äì167 (`stems = extract_stems(...)` and `new_fm = update_stems(...)`) and the `len(stems)` part of the print on line 171. The function signature and file read/write pattern stay unchanged.

- `claude/infra/spec-conventions.md` ‚Äî Phase 4 target: the `## Anchors` section (lines 33‚Äì35) references `{#slug}` format, heading-only placement, and the `stems:` frontmatter field. All three need updating. The `## Frontmatter` section (line 39) and the `## Template` code block (lines 52‚Äì53) also reference `stems:` as auto-generated ‚Äî remove that line.

- `claude/skills/write-spec/SKILL.md` ‚Äî Phase 4 target: step 3b (line 34) references `{#slug}` anchors with an implicit heading-only placement assumption. Step 3c (line 35) instructs running `spec-build-index` to regenerate both `features:` and `stems:`. The `spec-build-index` template block (lines 61‚Äì68) and `spec-format` anchoring rules (lines 99‚Äì103) describe the old stem system. The template frontmatter on lines 77‚Äì78 explicitly names `stems:` as auto-generated.

- `ai-docs/spec/skills.md` ‚Äî existing legacy stems: 14 stems in `file:dotted.path` format in the `stems:` frontmatter block (lines 19‚Äì33). All are non-dated `{#slug}` anchors in the body (`{#workflow-skills}`, `{#discussion}`, etc.). This file is the Phase 5 migration target and is already modified on the current branch (`M` in git status).

## Constraints

- `_replace_or_insert_block` currently inserts the block at the end of frontmatter if the key is absent. For Phase 3 "remove stems", the caller must skip the insert branch when `new_field_lines` is empty ‚Äî or add a dedicated `remove_block` path. The current function does not support removal natively.

- `extract_stems` (lines 45‚Äì93) builds dotted parent-chain keys using a `parent_slugs: dict[int, str]` depth stack. This logic is entirely stems-specific and has no reuse value after Phase 3 ‚Äî discard the entire function.

- The `SPEC_BASE` constant (line 20) in `spec-build-index` is used only by `extract_stems` to compute `file_stem` relative paths for the `file:dotted.path` key format. After removing `extract_stems`, `SPEC_BASE` becomes unused.

- All existing anchors in `ai-docs/spec/skills.md` are legacy format (`{#slug}`, no date prefix). The grep pattern `\{#\d{6}-` will return no matches against the current spec files until Phase 5 migration runs. The new `list-stems` must handle the zero-results case gracefully for both the no-arg and with-file-arg paths.

- The `generate-spec-stem` command needs the `ai-docs/spec/` directory path. `list-mental-model` resolves its base directory as a relative `Path('ai-docs/mental-model')` ‚Äî this works only when the CWD is the project root. `spec-build-index` resolves its base via `Path(__file__).resolve().parents[2]` (absolute from script location). `generate-spec-stem` should follow the `spec-build-index` approach: derive `ai-docs/spec/` from `__file__` to avoid CWD dependency.

- `write-spec/SKILL.md` step 3c currently names both `features:` and `stems:` as outputs of `spec-build-index`. After Phase 3, `spec-build-index` only regenerates `features:`. The step description and the `spec-build-index` template block (lines 61‚Äì68) both need updating to remove the `stems:` reference.
