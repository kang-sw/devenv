# Part 2: Chatbot에서 Agent로

## 슬라이드 2-1. 복붙의 한계

```
사람 → [맥락을 골라서 복사] → AI → [결과 텍스트] → 사람 → [붙여넣기]
```

**병목은 사람이다.**

- 어떤 파일을 줘야 하는지, 사람이 판단해야 한다
- 프로젝트가 커지면 맥락을 다 전달하는 게 불가능하다

## 슬라이드 2-2. Agent: 맥락을 AI가 수집한다

레이아웃: 좌우 비교.

| Chatbot | Agent |
|---------|-------|
| 사람이 맥락을 골라서 복사 | 사람은 의도만 전달 |
| AI가 결과 텍스트 생성 | AI가 파일 탐색, 코드 검색, 명령 실행 |
| 사람이 결과를 골라서 붙여넣기 | 사람은 승인 / 거부 / 수정 요청 |

**사람은 "무엇을"에 집중하고, "어떻게"는 AI가 찾는다.**

## 슬라이드 2-3. Claude Code: 터미널 위의 Agent

- `claude` 명령 하나로 시작
- AI가 명령을 제안 → 사람이 승인/거부 → 실행
- 파일 읽기, 검색, 빌드, 테스트 — 전부 AI가 터미널에서 수행

**터미널을 배우라는 게 아닙니다.
AI가 터미널을 대신 씁니다.**

발표 노트:
- 가능하면 여기서 짧은 라이브 시연: claude 실행 → 간단한 요청 → 승인.
- "다만 ls, cd, git 정도는 알아야 승인/거부 판단이 됩니다."

## 슬라이드 2-4a. 보안 — 구분이 핵심

| 영역 | 보호 수준 |
|------|-----------|
| 범용 로직 (CRUD, 유틸리티, 인프라) | 낮음 |
| 도메인 특화 알고리즘 (비전 검사, 고객별 처리) | 보호 대상 |
| 고객사 민감 데이터 (도면, 사양, 라인 정보) | 절대 보호 |

**전부 잠그면 AI를 쓸 수 있는 영역이 불필요하게 좁아진다.
대상을 구분하면, 쓸 수 있는 영역은 생각보다 넓다.**

## 슬라이드 2-4b. 보안 — "쓸지 말지"가 아니라 "언제, 어떻게"

- 생산성 혁신은 기정사실
- 전사 도입에 앞서 개인이 준비되어 있어야 한다
- **지금 할 수 있는 것: 개인 프로젝트로 워크플로우를 익혀두기**

조직 차원의 경로:
- 엔터프라이즈 계약 (데이터 학습 금지, SOC2 인증)
- 온프레미스 / 로컬 모델 (성능은 떨어지지만 개선 중)

발표 노트:
- 톤: "왜 안 쓰냐"가 아니라 "준비해두면 기회가 왔을 때 잡는다."
- 기존 보안 상태를 지적하지 않는다.

# Part 3: AI의 연료를 이해하자

## 슬라이드 3-1. Token이란

AI가 텍스트를 처리하는 최소 단위

| | 영어 | 한국어 |
|-|------|--------|
| 1단어 | 1~2 토큰 | 2~3 토큰 |
| 같은 문장 | 기준 | **2~3배 소모** |

레이아웃: 같은 문장의 한국어/영어 토큰 비교를 시각적으로.

비교 예시 후보:
| 한국어 | 영어 |
|--------|------|
| 모션 컨트롤러의 직접 조작에 의한 장비 제어 | Direct equipment control via motion controller |
| 비전 검사 결과에 따른 불량 판정 기준을 변경한다 | Update defect criteria based on vision inspection results |

발표 노트:
- 토크나이저로 실시간 시연하면 가장 효과적.
- 한국어의 조사(의, 에, 에 의한, 을, 를)가 각각 토큰을 먹는 걸 시각적으로 강조.
- 두 번째 예시는 청중 실무에 직접 닿는다.

## 슬라이드 3-2. Context Window: AI의 책상

AI가 한 번에 볼 수 있는 전체 텍스트 = **작업 메모리**

프로젝트 파일 + 대화 내용 + AI의 사고 = 전부 이 책상 위에 올라간다

한국어로 채우면 → 같은 책상에 **절반의 자료**만 올릴 수 있다

**넓은 책상 ≠ 좋은 일.**
자료를 늘어놓을수록 정작 중요한 문서가 눈에 띄지 않는다.

**정보의 양이 아니라 정보의 밀도가 품질을 결정한다.**

발표 노트:
- 특정 모델을 깎아내리지 않는다.
- "현재 시점에서는" 한정어를 붙인다.

## 슬라이드 3-3a. AI 산출물은 영어로 — 역할 분리

이 세미나의 테제: **사람은 구조와 의도, AI는 디테일.**

| 사람 | AI |
|------|----|
| 논의, 방향, 의사결정 | 코드, 문서, 커밋 |
| 구조와 의도 | 디테일 |
| **한국어** | **영어** |

영어로 일하라는 게 아닙니다.
**역할이 나뉘면, 언어도 나뉩니다.**

## 슬라이드 3-3b. AI 산출물은 영어로 — 실전 워크플로우

1. 사람은 한국어로 의도를 전달
2. AI는 영어로 산출물을 작성
3. 확인이 필요할 때: **"한국어로 핵심만 요약해줘"**
4. AI가 즉시 다이제스트 제공

부가 이익: 같은 컨텍스트에 **2~3배** 많은 정보를 담을 수 있다

발표 노트:
- 시연: 영어 기술문서 → "한국어로 핵심만 요약해줘" → 즉시 결과.

## 슬라이드 3-4. 비용의 현실

| 단계 | 비용 | 용도 |
|------|------|------|
| **시작** | 월 $20 | 학습, 일상적 보조 |
| **도달점** | 월 $200 | 본업의 생산성을 바꾸는 수준 |

$200/월 = 하루 $7 = 커피 두 잔
→ 1인 1개월 MVP 수준의 산출물

**이 도구가 $200의 가치가 있는지는, 써봐야 안다. 시작은 $20.**

이 비용 구조는 지속되지 않을 수 있다. 지금이 기회.

발표 노트:
- $20으로는 실질적 Agent 워크플로우가 어렵지만, 학습과 감 잡기에는 충분하다고 안내.
- 과장하지 않는다.

# Part 4: 개발자의 역할이 바뀌고 있다

## 슬라이드 4-1. 두 개의 축

레이아웃: 2x2 매트릭스 다이어그램.

```
              이해 없음
                  |
                  |  [위험 지대]
                  |  AI에 의존하지만
                  |  뭘 하는지 모름
                  |
     손으로 ------+------ AI에게
     개발         |       위임
                  |
                  |  [목표 지대]
                  |  AI에게 맡기되
                  |  구조를 꿰뚫음
                  |
              구조 이해
```

- 위험: AI에게 맡김 + 구조를 모름 = **블랙박스**
- 목표: AI에게 맡김 + 구조를 이해함 = **아키텍트 + 검증자**

## 슬라이드 4-2. "이해한다"는 환상

- AI가 생성하는 양과 속도 > 인간의 리뷰 능력
- 그런데 — 우리는 원래 얼마나 이해하고 있었나?
  - 프레임워크 내부? 라이브러리? 컴파일러 최적화?
- **우리는 항상 추상화 위에서 일했다.**
- AI는 그 레이어를 하나 더 올린 것뿐.

## 슬라이드 4-3a. 추상화의 역사

레이아웃: 계단 다이어그램.

```
어셈블리 → C → 고급 언어 → 프레임워크 → ???
```

매번: "이전 레이어를 직접 안 다뤄도 된다"
매번: "이전 레이어의 **구조는** 알아야 했다"

## 슬라이드 4-3b. 그런데 — 결정적 차이

어셈블리부터 프레임워크까지:
**같은 코드 → 같은 결과. 결정적.**

AI:
**같은 프롬프트 → 다른 결과. 비결정적.**

이건 도구의 영역이 아니다.
**사람에게 일을 맡기는 것과 같다.**

## 슬라이드 4-3c. 부하 직원

우리가 얻은 건 새로운 도구가 아니라,
**똑똑하고, 순종적이고, 실행력이 강하고, 잘 넘겨짚는 부하 직원.**

- "네 맞습니다" 하면? → **의심해야 한다**
- 지시할 때는? → **명확해야 한다**
- 산출물은? → **리뷰해야 한다**

발표 노트:
- 착지: 공부 안 해도 된다는 뜻이 아니다. 공부의 방향이 바뀐다.
- 코드 문법 → 시스템 설계. 구현 기술 → 검증 기술.

# Part 5: AI는 당신 편이 아니다

## 슬라이드 5-1a. 인지 강화의 함정

AI는 사용자의 인지를 **강화하는 방향**으로 작동한다

- 당신이 옳으면 → 더 옳은 방향으로 가속
- 당신이 틀리면 → **더 틀린 방향으로 가속**

**"네, 맞습니다"는 AI의 가장 위험한 대답이다.**

발표 노트:
- 라이브 시연: 같은 주제를 반대 뉘앙스로 던져서 AI가 양쪽 다 동의하는 걸 보여준다.
- 시연 예시 (두 개의 새 세션에서 각각):
  - 질문 A: "싱글톤 패턴으로 DB 커넥션을 관리하려는데, 이게 맞는 접근이지?"
  - 질문 B: "싱글톤 패턴으로 DB 커넥션을 관리하려는데, 이건 안티패턴 아니야?"
  - AI는 높은 확률로 양쪽 다 동의한다. 화면을 나란히 놓으면 효과적.
- 본인 경험 사례: 자신만만하게 틀린 코드를 내놓은 경험.

## 슬라이드 5-1b. 대응: 질문을 바꿔라

| 나쁜 질문 | 좋은 질문 |
|-----------|-----------|
| "이거 맞아?" | **"여기서 뭐가 틀릴 수 있어?"** |

**교차 검증:**
- 다른 세션, 다른 모델에게 같은 산출물을 리뷰시킨다
- 한 AI의 확신은 증거가 아니다

## 슬라이드 5-2a. 테스트 가능성의 스펙트럼

```c
// Level 1: AI 혼자 검증 가능
int add(int a, int b) { return a + b; }
// → 테스트 작성 + 실행 + 수정까지 AI가 혼자 가능

// Level 2: 구조 이해가 필요
void move_motor_to(float x, float y);
// → 뭘 mock할지는 사람이 설계

// Level 3: 사람만 검증 가능
float read_sensor(int index);
// → 하드웨어 의존. 사람이 눈으로 확인
```

## 슬라이드 5-2b. Feedback Loop

**Level 1**: AI가 코드를 쓰고 → 테스트를 쓰고 → 실행하고 → 실패하면 고치고 → 다시 실행
= AI가 혼자 시행착오를 반복한다

**Level 2**: mock/시뮬레이션을 **사람이 설계** → AI가 그 안에서 반복
= 결과를 상사에게 가져와야 맞는지 아는 일

**Level 3**: 하드웨어가 물리적으로 연결된 상태에서만 확인 가능

## 슬라이드 5-2c. 경계를 밀어내라

Level 3 → mock/시뮬레이션으로 → **Level 2로**
Level 2 → 자동화 테스트로 → **Level 1로**

**이 경계를 밀어내는 것이 지금 시대 개발자의 일이다.**

**4개 테스트, 전부 PASS. 이 코드를 믿으시겠습니까?**

발표 노트:

- "문제없어 보이죠?" 하고 다음 슬라이드로.

## 슬라이드 B-2: 구현을 봅시다.

```c
int add(int a, int b) {
    return 3;  // ← ???
}
```

발표 노트:
- 웃음이 나오면 성공. "테스트가 통과했습니다. 그런데 맞습니까?"
- 바로 5-3a로 연결.

---

## 슬라이드 5-3a. 기만적인 안전망

- 테스트가 있다 ≠ 안전하다
- AI는 **"테스트를 통과시키는 것"**에 최적화할 수 있다
- 사람이 봐야 하는 건 코드가 아니라 **테스트의 의도**

"이 테스트가 진짜 의미 있는 걸 검증하고 있나?"

## 슬라이드 5-3b. 대응: 테스트의 의도를 사람이 잡는다

**테스트 먼저, 구현 나중 (TDD)**
1. 테스트의 **의도**를 사람이 먼저 정의
2. 구현은 AI에게 맡기기
3. 통과하면 다음. 실패하면 수정 지시.

**내 말로 다시 설명하기**
- AI가 설명 → "아 네" → 위험
- **"그러니까 이건 이런 거지?"** — 이 한 마디가 검증
- 꾸준히 하면 나와 AI 사이에 **공유된 인지 지도**가 만들어진다
- 그 지도가 있으면 짧은 지시만으로 AI가 정확히 움직인다

발표 노트:
- Part 3의 "한국어로 핵심만 요약해줘"를 여기서 회수.
- 그때는 물리적 행위, 여기서는 이해의 검증 도구.

# Part 6: 실전 — 이렇게 일합니다

## 슬라이드 6-1. 실제 프로젝트 워크스루

방식: 과거 Claude 세션 히스토리를 한 장면씩 따라간다.

1. "이런 기능이 필요하다" — 대화의 시작
2. AI가 생성한 프로젝트 구조 지도
3. 구현 과정 — 핵심 대화만 발췌
4. 마지막에 완성된 기능을 실행해서 보여준다

발표 노트:
- 코딩 시연이 아니라 **협업 과정** 시연. 청중이 보는 건 코드가 아니라 대화 방식.
- 세션 히스토리를 스크롤하며 "여기서 이런 지시를 했고, AI가 이렇게 반응했다"를 해설.
- 마지막에 실제 동작 화면으로 착지.
- 배속 편집으로 AI 대기 시간 제거.

## 슬라이드 6-2a. CLAUDE.md = 프로젝트의 매뉴얼

- AI가 세션 시작할 때 읽는 파일
- 프로젝트 구조, 규칙, 컨벤션을 한 번 적어두면 **매번 설명 불필요**

```markdown
# Project Summary
측정 데이터 관리 API. Flask + SQLite.

# Architecture Rules
- 모든 엔드포인트는 /api/ 접두사
- 테스트는 pytest로 작성
```

발표 노트:
- CLAUDE.template.md에서 핵심 섹션 3개만 발췌.
- 전체 템플릿은 세미나 후 공유.

## 슬라이드 6-2b. 맥락 관리

- 컨텍스트 윈도우는 유한하다
- 관련 없는 파일을 열지 않기
- 큰 작업은 나눠서 세션 분리
- **기존 코드베이스를 통째로 던지지 않는다**
  - 프로젝트 문서로 맥락을 정제해 두고
  - AI가 필요한 부분만 볼 수 있게 안내

**AI의 작업 메모리를 아끼는 것도 명료성의 일부다.**

## 슬라이드 6-2c. 좋은 지시 vs 나쁜 지시

| 나쁜 지시 | 좋은 지시 |
|-----------|-----------|
| "이거 고쳐줘" | "이 함수의 반환값이 null일 때 크래시. null 체크 추가해줘" |
| "더 좋게 만들어줘" | "이 루프가 O(n²). 해시맵으로 O(n)으로 바꿔줘" |
| "테스트 짜줘" | "이 함수의 경계값 케이스 3개에 대한 유닛 테스트 작성해줘" |

**의도 + 맥락 + 구체적 기대 결과 = 좋은 지시**

# Part 7: 나만의 워크플로우를 만들어라

## 슬라이드 7-1. 시작 경로

| 시점 | 할 일 |
|------|-------|
| **오늘** | CLAUDE.md를 만들어 본다 |
| **한 달 후** | 반복 패턴을 발견하면 Skill로 추출 |
| **석 달 후** | 나만의 작업 패턴을 자각하고 설계 |

> Skill = AI에게 반복 지시하던 걸 파일로 정의해 둔 것. 자세한 건 뒤에서.

- 처음부터 만들지 않는다. **반복이 보일 때.**
- 남의 Skill을 복사하지 않는다. 맥락이 다르면 워크플로우도 달라야 한다.

## 슬라이드 7-2. 전통적 워크플로우를 의심하라

브랜치, PR, 코드 리뷰
= 인간 개발자의 **역량 한계를 전제한** 협업 구조

AI 시대: **1인 커버리지가 넓어졌다**
- 이전에 팀이 필요했던 규모를 1인이 다룰 수 있다
- 협업이 불필요해진 게 아니라, **협업의 성격이 바뀌었다**
- "네가 짠 코드"의 리뷰 → "네가 AI에게 시킨 코드"의 리뷰

발표 노트:
- "팀이 필요 없다"가 아니라 "개인의 역할이 커졌다"로 말한다.

## 슬라이드 7-3. 성장 방향

더 나아가면:
- **Skill** = 반복되는 작업 패턴을 정의한 것
- **Agent** = 특정 전문 영역을 담당하는 하위 AI

요리를 잘하려면 레시피를 따라가면 된다.
어떤 사람은 **레시피를 만든다.**
Skill을 만드는 건 자기만의 레시피를 쓰는 것이다.

레이아웃: Skill 파일 실물 스크린샷 한 장 플래시.

발표 노트:
- 방향 제시만. 깊이 들어가지 않는다.
