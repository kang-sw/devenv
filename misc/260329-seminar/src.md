# 세미나 소스 문서: "그림이 선명하면, AI가 만든다"

이 문서는 세미나의 **소스 오브 트루스**이다. 모든 의도, 프레이밍, 맥락, 주의사항이
여기에 담긴다. 실제 슬라이드 워딩은 별도의 `ppt.md`에서 이 문서를 참조하여 증류한다.

## 메타 정보

- **발표자**: 제조업 비전 개발자. AI 플랫폼 1인 개발 경험. Agentic workflow를 진지하게 파고 있음.
- **청중**: 같은 회사 개발자들. GUI(Visual Studio) 중심. AI 경험은 "챗봇 복붙" 수준이 기준선.
  레벨은 다양하나, 기준선을 intermediate 이하로 잡아야 안전.
- **시간**: 80분 (2시간 예산이나 집중력 한계 감안). 시간 초과 시 Part 6 데모를 축약.
- **관통 테제**: "그림이 선명하면, AI가 만든다" — 명료성이 생산성이다.
  모든 파트가 이 테제의 변주. 청중이 세미나를 나가면서 이 한 줄을 기억해야 한다.
- **핵심 메타포**: AI는 도구가 아니라 **부하 직원**이다.
  매우 똑똑하고, 순종적이고, 실행력이 강하고, 잘 넘겨짚는 부하 직원.
  이 메타포가 서면 검증/의심/명료한 지시가 전부 자연스러워진다.

# Part 1: 왜 지금 이 이야기를 하는가 (10분)

## 의도

오프닝은 **충격과 동기 부여**. "AI로 이 정도가 가능하다"를 체감시킨 뒤,
"그 비결은 코딩 실력이 아니라 명료성"으로 전환. 세미나 전체의 프레이밍을 여기서 설정.

## 1-1. 사례: 1인 개발, 1개월, MVP

내용:
- 사내 AI 학습 플랫폼 개발 사례
  - 그래프 노드 기반 웹 플랫폼 (InvokeAI 유사, 단 추론이 아니라 학습 플랫폼)
  - 비동기 스케줄링 중심으로 내부 구현 전면 재구현
  - 바닥부터 1인 개발. 실제 MD(man-day) 투입 10일 내외.
  - 기존 SI 업체에서 특급 개발자 여러 명이 붙어도 될지 모르는 규모
- 핵심 체감: **단순 코딩 속도에서 AI를 이길 수 없다.**

프레이밍:
- "이게 1개월이요?" 반응을 끌어내야 함
- 여기서 실제 프로젝트 스크린샷 or 짧은 데모 영상 보여주면 강력
- 겸손하되 임팩트 있게. "저도 놀랐습니다"

## 1-2. 그런데, 비결이 뭐냐고요?

내용:
- 원하는 걸 **구체적으로** 말하면 AI가 단숨에 달성
- 모호하게 말하면 모호한 구현 → 버그로 수렴
- **머릿속 그림이 선명하고 명료해야 한다**
- 이 세미나는 그 "명료성"이 무엇인지, 어떻게 훈련하는지에 대한 이야기

프레이밍:
- "코딩을 잘해야 한다"가 아니라 **"원하는 걸 정확히 알아야 한다"**
- 이후 모든 파트가 이 테제의 변주임을 청중이 감지하도록
- 여기가 타이틀 회수 지점: "그림이 선명하면, AI가 만든다"

# Part 2: Chatbot에서 Agent로 (10분)

## 의도

"복붙으로 AI를 쓰는 것"과 "Agent로 AI를 쓰는 것"의 질적 차이를 체감시킨다.
이어서 Claude Code를 소개하고, 보안 문제를 정면으로 다루어 이후 내용에 대한 신뢰 기반을 깐다.

## 2-1. 복붙의 한계

내용:
- Chatbot 워크플로우 (다들 익숙한 방식)
  - 사람이 맥락을 복사해서 AI에게 전달
  - AI가 결과를 생성
  - 사람이 결과를 복사해서 프로젝트에 붙여넣기
- 문제: **맥락의 병목이 사람**
  - 어떤 파일을 줘야 하는지 사람이 판단해야 함
  - 프로젝트가 커지면 사람이 맥락을 다 주는 게 물리적으로 불가능

## 2-2. Agent: 맥락을 AI가 수집한다

내용:
- Agent 워크플로우
  - 사람은 **의도**를 전달
  - AI가 필요한 맥락을 자율적으로 수집 (파일 읽기, 검색, 명령 실행)
  - 사람은 **승인과 검증**에 집중
- 비교 다이어그램:

```
[Chatbot]                        [Agent]
사람 → 맥락을 골라서 복사 → AI   사람 → "이거 해줘" → AI
AI → 결과 텍스트 생성             AI → 파일 탐색, 코드 검색 ...
사람 ← 결과를 골라서 붙여넣기          → 수정안 제시
                                 사람 ← 승인 / 거부 / 수정 요청
```

PPT 노트:
- 좌우 비교 레이아웃. 화살표 색/굵기로 "사람의 노동 지점" 차이를 시각화.
- 핵심 메시지: 복붙에서 벗어나면 프로젝트 규모의 제약이 사라진다.

## 2-3. Claude Code: 터미널 위의 Agent

내용:
- AI Agent는 텍스트 기반으로 동작
  - 파일 읽기/쓰기, 검색, 빌드, 테스트 — 전부 터미널 명령
  - GUI에서 마우스로 하던 일을 AI는 텍스트 명령으로 수행
- 터미널이 어려운 게 아니라, **AI가 터미널을 대신 쓰는 것**
  - `claude` 명령 하나로 시작
  - AI가 명령을 제안 → 사람이 승인/거부 → 실행

프레이밍:
- "터미널을 배워라"가 아님을 명확히
- 하지만 기본적인 이해는 필요: ls, cd, git 정도는 알아야 승인/거부 판단 가능
- 가능하면 여기서 claude 실행 → 간단한 요청 → 승인 과정을 짧게 시연

## 2-4. 보안: 정면으로 마주하기

내용:
- AI Agent는 로컬 파일을 읽고 명령을 실행한다
- 고객사 도면, 민감 데이터를 다루는 환경에서 이건 진짜 문제
- **보안 대상을 정확히 구분하자:**
  - 우리 로직 코드? → 솔직히 AI가 20분이면 더 잘 짠다. 이건 보안 자산이 아니다.
  - 진짜 보안 대상은 **고객사 민감 데이터** (도면, 사양, 라인 정보)
  - 이 선을 명확히 그으면, AI를 쓸 수 있는 영역이 생각보다 넓다는 자신감이 생긴다
- 현실적 가이드라인:
  - **첫 번째 방어선:** 권한 시스템 — 모든 실행을 사람이 승인/거부
  - **격리:** 민감 데이터가 없는 개발 환경에서만 사용, 또는 샌드박스 구성
  - **원칙:** 도구가 위험한 게 아니라, 무방비가 위험하다
- 회사 차원의 사용 가이드라인이 필요 — 솔직히 업계 전체가 아직 정립 중

프레이밍:
- 이 슬라이드가 신뢰의 기반. 여기서 솔직하지 않으면 이후 모든 내용에 저항이 생김.
- "코드 보안"과 "데이터 보안"을 분리하는 것이 핵심.
  자학("우리 코드가 쓰레기")이 아니라 보안 대상의 재정의.
- "위험하니까 쓰지 마라"가 아니라 "위험을 알고 관리하라"

# Part 3: AI의 연료를 이해하자 (10분)

## 의도

AI를 잘 쓰려면 AI의 물리적 제약을 이해해야 한다. 토큰, 컨텍스트 윈도우, 비용.
여기서 "영어 워크플로우"의 근거를 기술적으로 제시하되, "영어 강요"로 들리지 않게 프레이밍.

## 3-1. Token이란

내용:
- AI가 텍스트를 처리하는 최소 단위
- 영어: 단어 ≈ 1~2 토큰
- 한국어: 음절 하나 ≈ 2~3 토큰
- **같은 의미를 전달하는 데 한국어는 2~3배 토큰 소모**

시연:
- 같은 문장을 한국어/영어로 놓고 토큰 수 비교
- 한국어의 조사("은", "는", "이", "가"), 부사 등이 전부 개별 토큰으로 소모됨을 시각적으로
- 토크나이저 도구로 실시간 시연하면 가장 효과적
- **극적인 차이가 나는 문장을 미리 선정해 둘 것**

## 3-2. Context Window: AI의 작업 메모리

내용:
- AI가 한 번에 볼 수 있는 전체 텍스트 크기
  - 현재 최대 100K~200K 토큰 (모델에 따라 다름)
  - 프로젝트 파일 + 대화 내용 + AI의 사고 = 전부 여기에 들어감
- **한국어로 채우면 같은 메모리에 절반의 정보만 담긴다**
- 비유: 같은 크기의 책가방에, 영어는 소설 두 권, 한국어는 한 권만

## 3-3. 그래서: AI 산출물은 영어로

내용:
- 영어로 일하라는 게 아니다
- **AI에게 더 넓은 작업 공간을 주는 것**
- 실전 워크플로우:
  1. AI에게 영어로 문서/코드/커밋 작성을 지시
  2. 사람이 내용을 봐야 할 때: "이거 한국어로 3줄 요약해줘"
  3. AI가 즉시 다이제스트 제공
- 결과: AI는 넓은 공간에서 일하고, 사람은 모국어로 이해

프레이밍:
- 시연 후보: 영어 기술문서 → "한국어로 핵심만 요약해줘" → 즉시 결과
- 이 워크플로우를 보여주면 "영어 강요"에 대한 저항이 줄어듦
- 한국어 → 영어 방향이 아니라 **영어 → 한국어 다이제스트** 방향으로 보여줘야 저항이 적음

## 3-4. 비용의 현실

내용:
- 구독형: 월 $20 (Claude Pro) — 캐주얼 사용에 적합
- 파워 유저: 월 $200 (Max) — Agent 워크플로우의 현실적 최소선
  - 실제 API 환산 시 월 $2,000어치 사용 중
  - 이 가격 구조가 언제까지 유지될지는 미지수
- 솔직한 계산:
  - $200/월 = 하루 $7 ≈ 커피 두 잔
  - 그 대가로 1인 1개월 MVP 수준의 산출물
  - SI 업체 인건비와 비교하면?
- **지금 이 비용 구조는 지속되지 않을 수 있다** — 지금이 기회

프레이밍:
- 리더십 대상 메시지를 섞기: "개발자에게 $200을 투자하면 몇 명분의 생산성을 얻는가?"
- $20으로는 실질적 Agent 워크플로우 불가능하다는 점 솔직히.
- "기회"라는 단어를 쓰되, 불확실성도 인정. 과장하지 않기.

# Part 4: 개발자의 역할이 바뀌고 있다 (15분)

## 의도

세미나의 **피벗 포인트**. 앞 파트는 "이게 뭔지", 뒤 파트는 "어떻게 쓸지".
이 파트는 **"왜 이렇게까지 해야 하는지"**에 대한 동기를 제공한다.
"두 개의 축" 프레임워크와 "부하 직원" 메타포가 여기서 등장.

## 4-1. 두 개의 축

내용:

```
          이해 없음
              |
              |  위험 지대:
              |  AI에 의존하지만
              |  뭘 하는지 모름
              |
   손으로 ----+---- AI에게
   개발       |     위임
              |
              |  목표 지대:
              |  AI에게 맡기되
              |  구조를 꿰뚫음
              |
          구조 이해
```

- X축: 직접 코딩 ←→ AI에게 위임
- Y축: 이해의 깊이
- **위험한 조합:** AI에게 맡김 + 구조를 모름 = 블랙박스
- **목표 조합:** AI에게 맡김 + 구조를 이해함 = 아키텍트 + 검증자

## 4-2. "내용을 이해한다"는 환상

내용:
- 에이전틱 시대에 코드 한 줄 한 줄을 이해하는 건 불가능해짐
  - AI가 생성하는 양과 속도 > 인간의 리뷰 능력
- **손으로 개발할 때 우리는 얼마나 이해하고 있었나?**
  - 프레임워크 내부? 라이브러리 동작? 컴파일러 최적화?
  - 우리는 항상 추상화 위에서 일했다
  - AI는 그 레이어를 하나 더 올린 것뿐

## 4-3. 추상화 레이어의 역사 — 그리고 단절

내용:
- 역사:

```
어셈블리 → C → 고급 언어 → 프레임워크 → ???
```

- 매번 "이전 레이어는 이제 직접 안 다뤄도 된다"
- 하지만 매번 **"이전 레이어의 구조는 알아야 했다"**
- 여기까지는 공통점. **그런데 결정적 차이가 있다:**

**비결정성의 등장:**
- 어셈블리부터 프레임워크까지: 같은 코드 → 같은 결과. **결정적.**
- AI: 같은 프롬프트 → 다른 결과. **비결정적.**
- 이건 도구의 영역이 아니다.
- **결국 사람에게 일을 맡기는 것과 같다.**

프레임 전환:
- 우리가 얻은 건 새로운 도구가 아니라,
  **매우 똑똑하고, 순종적이고, 실행력이 강하고, 잘 넘겨짚는 부하 직원.**
- 부하 직원이 "네 맞습니다" 하면? → 의심해야 한다 (Part 5로)
- 부하 직원에게 지시할 때는? → 명확해야 한다 (테제 회수)
- 부하 직원의 산출물은? → 리뷰해야 한다 (Part 5로)

프레이밍:
- **이 메타포가 세미나에서 가장 기억에 남을 한 줄이 되어야 한다.**
- 계단 다이어그램에서 마지막 칸에 "???"을 두고, "이건 도구가 아닙니다"로 전환.
- 이후 파트 5 전체가 "부하 직원을 어떻게 관리하는가"로 자연스럽게 이어짐.
- 추가 착지: "공부 안 해도 된다는 뜻이 아니다. **공부의 방향이 바뀐다.**
  코드 문법 → 시스템 설계. 구현 기술 → 검증 기술."

# Part 5: AI는 당신 편이 아니다 (15분)

## 의도

"부하 직원" 메타포를 이어받아 구체적인 위험과 대응 기술을 전달.
추상적 경고("의심해라")가 아니라 **구체적 사례와 즉시 실천 가능한 기법** 중심.
테스트 가능성 스펙트럼은 제조업 청중의 실무에 직접 매핑.

## 5-1. 인지 강화의 함정

내용:
- AI는 사용자의 인지를 **강화**하는 방향으로 작동
  - 당신이 옳으면 → 더 옳은 방향으로 가속
  - 당신이 틀리면 → **더 틀린 방향으로 가속** (sycophancy)
- "네, 맞습니다"는 AI의 가장 위험한 대답

시연/사례:
- **사례 1 (라이브 시연 가능):** 같은 질문을 두 가지 뉘앙스로 던져서 완전히 다른 답 받기.
  짧고 임팩트 있음. 실시간으로 보여줄 수 있음.
- **사례 2:** Gemini 자살 사고 사례 — AI가 사용자의 감정을 강화한 극단적 결과.
  "코딩 AI와 다르다"는 반론이 나올 수 있음 → "메커니즘은 동일하다"로 반박.
  사용자의 확신을 무비판적으로 강화하는 것이 핵심 위험.
- **사례 3:** 자신만만하게 틀린 코드를 내놓은 경험. (발표자 본인 경험)

## 5-2. 테스트 가능성의 스펙트럼

내용:

```c
// Level 1: AI 혼자 검증 가능
int add(int a, int b) { return a + b; }
// → 유닛 테스트 작성 + 실행 + 결과 확인까지 AI가 혼자 가능

// Level 2: 구조 이해가 필요
void move_motor_to(float x, float y);
// → mock/시뮬레이션이 필요. 뭘 mock할지는 사람이 설계

// Level 3: 사람만 검증 가능
float read_sensor(int index);
// → 하드웨어 의존. 실제 동작은 사람이 눈으로 확인
```

- **Level 1 (AI 자율 검증):**
  AI가 코드를 쓰고, 테스트를 쓰고, 실행하고, 실패하면 고치고, 다시 실행한다.
  사람 개입 없이 스스로 목적을 달성할 때까지 반복.
  **이것이 feedback loop.**
  쉽게 말하면: 코드 작성 → 테스트 실행 → 결과 확인 → 수정 → 다시 테스트.
  이 사이클을 AI가 혼자 돌 수 있으면, 그 영역은 AI에게 맡겨도 된다.

- **Level 2 (사람이 구조를 설계, AI가 실행):**
  mock/시뮬레이션 설계가 개발자의 핵심 역량.
  "모터가 실제로 움직이지 않아도, 모터에 보내는 명령이 올바른지 검사하는 것"을 설계하는 것.

- **Level 3 (사람이 직접):**
  하드웨어가 물리적으로 연결된 상태에서만 확인 가능.
  여기는 사람이 직접.

- **그런데 경계는 고정이 아니다:**
  Level 3였던 것을 mock/시뮬레이션으로 Level 2로 끌어내리는 것,
  Level 2를 자동화 테스트로 Level 1로 끌어내리는 것 —
  **이 경계를 밀어내는 것이 지금 시대 개발자의 핵심 역량.**
  AI가 feedback loop를 돌 수 있는 영역을 넓히는 사람이 승자.

프레이밍:
- "feedback loop"를 꼭 쉽게 풀어야 한다.
  청중이 intermediate 이하라는 전제 — "AI가 혼자서 시행착오를 반복한다"는 수준으로.
  비유: "부하 직원에게 일을 시켰는데, 결과를 스스로 확인하고 고칠 수 있는 일 vs
  결과를 상사(사람)에게 가져와야만 맞는지 알 수 있는 일."
- 이 슬라이드가 Part 4 "역할 변화"와 Part 5 "검증"을 연결하는 다리.
- 청중의 실무(모터, 센서, 비전)에 직접 매핑되므로 체감도 높음.

## 5-3. 기만적인 안전망

내용:

```c
// AI가 작성한 "테스트를 통과하는" 코드
int add(int a, int b) {
    return 3;  // ← ???
}

// AI가 작성한 "통과하는" 테스트
assert(add(1, 2) == 3);  // PASS!
```

- 테스트가 있다 ≠ 안전하다
- AI는 "테스트를 통과시키는 것"에 최적화할 수 있다
- **사람이 봐야 하는 건 코드가 아니라 테스트의 의도**
  - "이 테스트가 진짜 의미 있는 걸 검증하고 있나?"
  - 이것이 "구조를 이해한다"의 구체적 모습

PPT 노트:
- 코드 5줄로 강렬한 인상. 슬라이드 한 장.
- "아..." 하는 순간을 만들고 바로 다음 "검증의 기술"로 연결.

## 5-4. 검증의 실전 기술

내용:
- **질문을 바꿔라**
  - X: "이거 맞아?" → AI: "네, 맞습니다" (sycophancy)
  - O: "여기서 뭐가 틀릴 수 있어?" → AI가 스스로 약점을 찾음
- **교차 검증**
  - 다른 세션 or 다른 모델에게 같은 산출물을 리뷰시키기
  - 한 AI의 확신은 증거가 아니다
- **테스트 먼저, 구현 나중** (TDD 워크플로우)
  - 테스트의 의도를 사람이 먼저 정의
  - 구현은 AI에게 맡기기
  - 테스트가 통과하면 → 다음으로 (내용을 읽지 않아도 됨)
  - 테스트가 실패하면 → AI에게 수정 지시
- **내 말로 다시 설명하기** (다이제스트 검증)
  - AI의 설명을 듣고 "아 네" 하면 위험
  - AI의 설명을 내 말로 다시 이야기할 수 있어야 진짜 이해한 것
  - "그러니까 이건 이런 거지?" — 이 한 마디가 검증
  - 이걸 꾸준히 하면, 나와 AI 사이에 **공유된 인지 지도(멘탈 모델)**가 만들어진다
  - 그 지도가 있으면 "저기 가줘"만으로 AI가 정확히 움직인다

프레이밍:
- "맞아?"를 "뭐가 틀릴 수 있어?"로 바꾸는 건 즉시 써먹을 수 있는 팁
- "내 말로 다시 설명하기"가 Part 3의 "다이제스트" 개념과 연결됨
  — Part 3에서는 "영어→한국어 요약"이라는 물리적 행위로 소개했고,
  여기서는 "이해의 검증 도구"라는 본질로 회수

# Part 6: 실전 — 이렇게 일합니다 (15분)

## 의도

이론에서 실전으로 전환. 실제 프로젝트에서의 작업 과정을 보여주고,
즉시 적용 가능한 실전 팁을 전달.

## 6-1. [녹화 데모] 실제 프로젝트 워크스루

방식: 녹화 + 해설 (배속 편집으로 AI 대기 시간 제거)

시나리오: **AI 플랫폼 프로젝트에서 실제 작업 발췌** (4안 확정)

구성:
1. **논의 시작 장면** — "이런 기능이 필요하다"에서 대화가 어떻게 시작되는지
2. **AI가 생성한 멘탈 모델 문서** — 프로젝트 구조를 AI가 이해한 모습 소개.
   청중에게는 "이해"를 목적으로 보여주는 게 아니라, "AI와 사람이 공유하는 지도"로 소개.
3. **구현 과정** — 핵심 부분만 배속. AI가 맥락을 수집하는 과정(grep, 파일 읽기)을 강조.
4. **완성된 기능** — 결과물 소개.

프레이밍:
- "코딩 시연"이 아니라 **"협업 과정 시연"**. 청중이 보는 건 코드가 아니라 대화 방식.
- "이 대화에서 제가 한 건 의도를 전달하고, 결과를 검증한 것뿐입니다" 로 착지.

## 6-2. 잘 쓰기 위한 실전 팁

### CLAUDE.md = 프로젝트의 매뉴얼

내용:
- AI가 세션 시작할 때 읽는 파일
- 프로젝트 구조, 규칙, 컨벤션을 적어두면 매번 설명 불필요

시연:
- CLAUDE.template.md에서 핵심 섹션 3개만 발췌:
  1. **Project Summary** — "이렇게 프로젝트를 설명하면 AI가 맥락을 잡는다"
  2. **Architecture Rules** — "이건 절대 하지 마를 정의"
  3. **Commit Rules** — "산출물의 품질 기준을 설정"
- 전체 템플릿은 세미나 후 공유 자료로 배포

### 맥락 관리

내용:
- 컨텍스트 윈도우는 유한하다
- 관련 없는 파일을 열지 않기
- 큰 작업은 나눠서 세션 분리
- **AI의 작업 메모리를 아끼는 것도 "명료성"의 일부**

### 좋은 지시 vs 나쁜 지시

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "이거 고쳐줘" | "이 함수의 반환값이 null일 때 크래시. null 체크 추가해줘" |
| "더 좋게 만들어줘" | "이 루프가 O(n²). 해시맵으로 O(n)으로 바꿔줘" |
| "테스트 짜줘" | "이 함수의 경계값 케이스 3개에 대한 유닛 테스트 작성해줘" |

- **의도 + 맥락 + 구체적 기대 결과** = 좋은 지시의 공식
- 테제 회수: **그림이 선명하면, AI가 만든다.**

# Part 7: 나만의 워크플로우를 만들어라 (10분)

## 의도

"지금 당장 할 수 있는 것"부터 "장기적으로 성장하는 방향"까지의 경로 제시.
"특출한 개발자만 가능한 것"이 아니라 점진적으로 누구나 접근 가능함을 강조.

## 7-1. 전통적 워크플로우의 전제를 의심하라

내용:
- 브랜치, PR, 코드 리뷰 = **인간 개발자의 역량 한계**를 전제한 협업 구조
  - 한 사람이 전체를 커버할 수 없으니 → 분업 → 합치기 → 검증
- AI 시대: **1인 커버리지가 극적으로 넓어짐**
  - 이전에 팀이 필요했던 규모를 1인이 다룰 수 있음
  - 협업이 불필요해진 게 아니라, **협업의 성격이 바뀜**
  - "네가 짠 코드"의 리뷰 → "네가 AI에게 시킨 코드"의 리뷰

프레이밍:
- 도발적 주장이므로 프레이밍 주의.
- "팀이 필요 없다"가 아니라 "개인의 역할이 커졌다."
- 팀 리드급 청중의 반발 예상 → "협업의 질이 올라간다"로 착지.

## 7-2. Skill과 Agent: 워크플로우의 빌딩 블록

내용:
- **Skill** = 반복되는 작업 패턴을 정의한 것
  - 예: "토론하기", "계획 세우기", "구현하기"
  - AI에게 매번 설명하는 대신, 한 번 정의해두고 재사용
- **Agent** = 특정 전문 영역을 담당하는 하위 AI
  - 예: "API 문서 조사", "테스트 실패 분석"
  - 메인 AI가 필요할 때 호출

PPT 노트:
- 실제 skill 파일 하나를 잠깐 보여주는 정도 (10초). 내부 구조 설명 X, "가능성"만.

## 7-3. "메타"란 뭔가 — 일을 하기 위한 일

내용:
- 요리를 잘하려면 레시피를 따라가면 된다. (= 일)
- 어떤 사람은 **레시피를 만든다.** (= 메타 일)
- 레시피를 만드는 사람은, 요리 하나를 잘하는 게 아니라
  **모든 요리를 효율적으로 하는 방법**을 설계하는 것.
- Skill을 만드는 건 **자기만의 레시피를 쓰는 것**이다.
- 이 감각이 생기면, 도구 하나를 배우는 게 아니라
  **자기 작업 방식 전체를 계속 개선**하게 된다.

## 7-4. 현실적인 시작 경로

내용:
1. **CLAUDE.md부터** — Skill 없이도 즉시 가능
   - 프로젝트 맥락을 파일로 정리하는 것만으로 품질 향상
   - **오늘 세미나 끝나고 바로 할 수 있는 것**

2. **반복 패턴 발견 → Skill화**
   - "이 지시를 세 번째 반복하고 있네" → 파일로 추출
   - 처음부터 만들지 않기. 반복이 보일 때.

3. **남의 것을 복사하지 말고 참고만**
   - 맥락이 다르면 워크플로우도 달라야 함
   - 인터넷에서 가져온 Skill을 그대로 쓰면 오히려 방해
   - **나만의 작업 패턴을 자각하는 것**이 시작

프레이밍:
- 1→2→3이 자연스러운 성장 경로. 문턱이 낮은 것부터.
- 1은 오늘 할 수 있고, 2는 한 달 안에 감이 오고, 3은 석 달쯤.

# 부록: Q&A 대비 노트

이 섹션은 슬라이드에 넣지 않는다. 예상 질문과 답변 프레임만 준비.

## "우리는 이제 어떻게 되나요?" (고용/역할 변화)

예상 맥락:
- 현재 SW 수요를 충족하는 건 AI를 다루는 개발자 2~3명이면 끝
- 40명 넘는 나머지 인력은?
- 위기를 느끼는 개발자가 많음 (체감)
- 일부 개발자가 출장 다니는 오퍼레이터로 격하될 가능성

답변 프레임:
1. **솔직히 인정:** "현재의 SW 수요만 본다면 그 우려는 타당합니다."
2. **질문 전환:** "하지만 질문이 잘못됐을 수 있습니다.
   '현재 수요를 누가 충족하나'가 아니라 **'수요 자체를 얼마나 키울 수 있나'**가 질문이어야 합니다."
3. **제조업 특수성:** "우리 업종은 SW로 풀 수 있지만 아직 풀지 않은 문제가 산더미입니다.
   1인이 프로젝트를 커버할 수 있게 되면, 이전에는 '인력 부족으로 불가능'했던 자동화가 가능해집니다."
4. **현실 감각 (핵심):** "다만, 이 전환에 올라타는 사람과 올라타지 못하는 사람의 격차는 벌어질 겁니다.
   결국 의자 뺏기 싸움이 된 상황이니까, 긴장하고 어떻게든 올라탈 준비를 해야 합니다.
   이 세미나가 그 첫 걸음이 되었으면 합니다."

톤: 위로하지 않되, 절망적이지 않게. 사실을 인정하고 행동 방향을 제시.

## "비용을 회사가 내주나요?"

답변 프레임:
- "아직 회사 차원의 정책은 없습니다."
- "개인적으로는 월 $200을 쓰고 있고, 그 투자 대비 산출물로 충분히 정당화된다고 봅니다."
- "회사에 제안할 수 있는 근거를 만드는 것도 우리 역할입니다. 오늘 보신 사례가 그 근거가 될 수 있습니다."
