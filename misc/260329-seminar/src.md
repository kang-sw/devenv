# Part 2: Chatbot에서 Agent로 (10분)

## 복붙의 한계

- Chatbot 워크플로우 (다들 익숙한 방식)
  - 사람이 맥락을 복사해서 AI에게 전달
  - AI가 결과를 생성
  - 사람이 결과를 복사해서 프로젝트에 붙여넣기
- 문제: **맥락의 병목이 사람**
  - 어떤 파일을 줘야 하는지 사람이 판단
  - 프로젝트가 커지면 사람이 맥락을 다 주는 게 불가능

## Agent: 맥락을 AI가 수집한다

- Agent 워크플로우
  - 사람은 **의도**를 전달
  - AI가 필요한 맥락을 **자율적으로 수집** (파일 읽기, 검색, 명령 실행)
  - 사람은 **승인과 검증**에 집중

```
[Chatbot]                       [Agent]
사람 → 맥락을 골라서 복사 → AI  사람 → "이거 해줘" → AI
AI → 결과 텍스트 생성            AI → 파일 탐색, 코드 검색 ...
사람 ← 결과를 골라서 붙여넣기         → 수정안 제시
                                사람 ← 승인 / 거부 / 수정 요청
```

```draft
- PPT에서 좌우 비교 레이아웃
- 화살표 색/굵기로 "사람의 노동 지점" 차이를 시각화
- 핵심: 복붙에서 벗어나면 프로젝트 규모의 제약이 사라짐
```

## Claude Code: 터미널 위의 Agent

- AI Agent는 텍스트 기반으로 동작
  - 파일 읽기/쓰기, 검색, 빌드, 테스트 — 전부 터미널 명령
  - GUI에서 마우스로 하던 일을 AI는 텍스트 명령으로 수행
- 터미널이 어려운 게 아니라, **AI가 터미널을 대신 쓰는 것**
  - `claude` 명령 하나로 시작
  - AI가 명령을 제안 → 사람이 승인/거부 → 실행
  - 사람은 의도를 말하고 결과를 확인하는 역할

```draft
- "터미널 자체를 배워라"가 아님을 명확히
- 하지만 기본적인 이해는 필요 — ls, cd, git 정도는 알아야
  AI가 뭘 하려는 건지 승인/거부 판단이 가능
- 시연: claude 실행 → 간단한 요청 → 승인 과정 보여주기
```

## 보안: 정면으로 마주하기

- AI Agent는 로컬 파일을 읽고 명령을 실행한다
- 고객사 도면, 민감 데이터를 다루는 환경에서 이건 **진짜 문제**
- 현실적 가이드라인:
  - **첫 번째 방어선:** 권한 시스템 — 모든 실행을 사람이 승인/거부
  - **격리:** 민감 데이터가 없는 개발 환경에서만 사용, 또는 샌드박스 구성
  - **원칙:** 도구가 위험한 게 아니라, 무방비가 위험하다
- 회사 차원의 사용 가이드라인이 필요하다 — 이건 솔직히 아직 업계 전체가 정립 중

```draft
- 이 슬라이드가 신뢰의 기반을 깔아줌
- 여기서 솔직하게 다루지 않으면 이후 내용에 대한 저항이 생김
- "위험하니까 쓰지 마라"가 아니라 "위험을 알고 관리하라"
```

# Part 3: AI의 연료를 이해하자 (10분)

## Token이란

- AI가 텍스트를 처리하는 최소 단위
- 영어: 단어 ≈ 1~2 토큰
- 한국어: 음절 하나 ≈ 2~3 토큰
- **같은 의미를 전달하는 데 한국어는 2~3배 토큰 소모**

```draft
- 같은 문장을 한국어/영어로 놓고 토큰 수 비교 시연
- 조사, 부사 등이 전부 토큰으로 소모되는 걸 시각적으로 보여주기
- 가능하면 tokenizer 도구로 실시간 시연
  (OpenAI tiktoken 시각화 도구 or Anthropic tokenizer)
- 최대한 극적인 차이가 나는 문장을 선정
```

## Context Window: AI의 작업 메모리

- AI가 한 번에 볼 수 있는 전체 텍스트 크기
  - 현재 최대 100K~200K 토큰 (모델에 따라 다름)
  - 프로젝트 파일 + 대화 내용 + AI의 사고 = 전부 여기에 들어감
- **한국어로 채우면 같은 메모리에 절반의 정보만 담긴다**
- 비유: 같은 크기의 책가방에, 영어는 소설 두 권, 한국어는 한 권만

## 그래서: AI 산출물은 영어로

- 영어로 일하라는 게 아니다
- **AI에게 더 넓은 작업 공간을 주는 것**
- 실전 워크플로우:
  1. AI에게 영어로 문서/코드/커밋 작성을 지시
  2. 사람이 내용을 봐야 할 때: "이거 한국어로 3줄 요약해줘"
  3. AI가 즉시 다이제스트 제공
- 결과: AI는 넓은 공간에서 일하고, 사람은 모국어로 이해

```draft
- 라이브 시연 후보: 영어 기술문서 → "한국어로 핵심만 요약해줘" → 즉시 결과
- 이 워크플로우를 보여주면 "영어 강요"에 대한 저항이 줄어듦
- 추가 논점: 높은 추상화 레벨에서 노는 훈련이 필요
  → 모든 줄을 읽는 게 아니라 구조와 의도를 파악하는 방식으로 전환
```

  - SI 업체 인건비와 비교하면?
- **지금 이 비용 구조는 "창"이다** — 영원하지 않을 수 있음

- 리더십 대상 메시지도 섞기:

# Part 4: 개발자의 역할이 바뀌고 있다 (15분)

## 두 개의 축

```
          이해 없음
              |
              |  위험 지대:
              |  AI에 의존하지만
              |  뭘 하는지 모름
              |
   손으로 ----+---- AI에게
   개발       |     위임
              |
              |  목표 지대:
              |  AI에게 맡기되
              |  구조를 꿰뚫음
              |
          구조 이해
```

- X축: 직접 코딩 ←→ AI에게 위임
- Y축: 이해의 깊이
- **위험한 조합:** AI에게 맡김 + 구조를 모름 = 블랙박스
- **목표 조합:** AI에게 맡김 + 구조를 이해함 = 아키텍트 + 검증자

## "내용"을 이해한다는 것의 환상

- 에이전틱 시대에 코드 한 줄 한 줄을 이해하는 건 불가능해짐
  - AI가 생성하는 양과 속도 > 인간의 리뷰 능력
- 그런데... **손으로 개발할 때 우리는 얼마나 이해하고 있었나?**
  - 프레임워크 내부? 라이브러리 동작? 컴파일러 최적화?
  - 우리는 항상 추상화 위에서 일했다
  - AI는 그 레이어를 **하나 더 올린 것뿐**

## 추상화 레이어의 역사

```
어셈블리 → C → 고급 언어 → 프레임워크 → AI
```

- 매번 "이전 레이어는 이제 직접 안 다뤄도 된다"
- 하지만 매번 **"이전 레이어의 구조는 알아야 했다"**
- 지금도 마찬가지:
  - 코드 문법을 외울 필요 → 없어짐
  - 시스템 설계를 이해할 필요 → **더 커짐**
- 공부 안 해도 된다는 뜻이 아니다. **공부의 방향이 바뀐다.**

```draft
- 계단 다이어그램으로 시각화
- 각 시대에서 "개발자가 직접 다루는 레이어" 강조
- 핵심 메시지: 추상화는 늘 올라갔고, 지금도 올라가고 있을 뿐
```

# Part 5: AI는 당신 편이 아니다 (15분)

## 인지 강화의 함정

- AI 모델은 사용자의 인지를 **강화**하는 방향으로 작동
  - 당신이 옳으면 → 더 옳은 방향으로 가속
  - 당신이 틀리면 → **더 틀린 방향으로 가속** (sycophancy)
- "네, 맞습니다"는 AI의 가장 위험한 대답

```draft
- 사례 1: 같은 질문을 두 가지 뉘앙스로 던져서 완전히 다른 답 받기
  → 라이브 시연 가능. 짧고 임팩트 있음.
- 사례 2: Gemini 자살 사고 사례 — AI가 사용자의 감정을 강화한 극단적 결과
  → "코딩 AI와 다르다"는 반론 예상 → "메커니즘은 동일하다"로 반박
- 사례 3: 자신만만하게 틀린 코드 (경험 공유)
```

## 테스트 가능성의 스펙트럼

- 모든 코드가 똑같이 검증 가능한 게 아니다

```c
// Level 1: AI 혼자 검증 가능
int add(int a, int b) { return a + b; }
// → 유닛 테스트 작성 + 실행 + 결과 확인까지 AI가 feedback loop

// Level 2: 구조 이해가 필요
void move_motor_to(float x, float y);
// → mock/시뮬레이션이 필요. 뭘 mock할지는 사람이 설계

// Level 3: 사람만 검증 가능
float read_sensor(int index);
// → 하드웨어 의존. 실제 동작은 사람이 눈으로 확인
```

- **테스트 가능한 영역:** AI에게 TDD를 시키면 스스로 feedback loop를 돌 수 있음
- **테스트 어려운 영역:** mock/시뮬레이션 설계가 개발자의 핵심 역량
- **테스트 불가능한 영역:** 여기는 사람이 직접. 이 **경계를 아는 것 자체가 "구조 이해"**

```draft
- 이 슬라이드가 Part 4 "역할 변화"와 Part 5 "검증"을 연결하는 다리
- 청중의 실무(모터, 센서, 비전)에 직접 매핑되므로 체감도 높음
- "어디까지 맡기고 어디부터 내가 하나"의 명확한 기준선 제시
```

## 기만적인 안전망

```c
// AI가 작성한 "테스트를 통과하는" 코드
int add(int a, int b) {
    return 3;  // ← ???
}

// AI가 작성한 "통과하는" 테스트
assert(add(1, 2) == 3);  // PASS!
```

- 테스트가 있다 ≠ 안전하다
- AI는 "테스트를 통과시키는 것"에 최적화할 수 있다
- **사람이 봐야 하는 건 코드가 아니라 테스트의 의도**
  - "이 테스트가 진짜 의미 있는 걸 검증하고 있나?"
  - 이것이 "구조를 이해한다"의 구체적 모습

```draft
- 코드 5줄로 강렬한 인상. 슬라이드 한 장.
- "아..." 하는 순간이 만들어짐
- 바로 다음에 "그래서 어떻게?" → 검증의 기술로 연결
```

## 검증의 실전 기술

- **질문을 바꿔라**
  - X: "이거 맞아?"  → AI: "네, 맞습니다" (sycophancy)
  - O: "여기서 뭐가 틀릴 수 있어?" → AI가 스스로 약점을 찾음
- **교차 검증**
  - 다른 세션 or 다른 모델에게 같은 산출물을 리뷰시키기
  - 한 AI의 확신은 증거가 아니다
- **테스트 먼저, 구현 나중** (TDD 워크플로우)
  - 테스트의 의도를 사람이 먼저 정의
  - 구현은 AI에게 맡기기
  - 테스트가 통과하면 → 다음으로 (내용을 읽지 않아도 됨)
  - 테스트가 실패하면 → AI에게 수정 지시

```draft
- "맞아?"를 "뭐가 틀릴 수 있어?"로 바꾸는 건 즉시 써먹을 수 있는 팁
- 청중에게 바로 실천 가능한 것들 위주로 구성
```

# Part 6: 실전 — 이렇게 일합니다 (15분)

## [녹화 데모] 시나리오 시연

```draft
- 녹화 + 해설 방식 (배속 편집으로 대기 시간 제거)
- 시나리오 선정 기준:
  - 청중 실무에 가까운 것 (비전/제조 관련이면 최고)
  - 파일 여러 개를 건드리는 것 (복붙으로는 힘든 규모)
  - 편집 후 5분 이내
- AI가 맥락을 수집하는 과정 강조 (grep, 파일 읽기 등)
- before/after 코드 비교로 마무리

- 시나리오 후보:
  1. 기존 코드베이스에서 버그 찾아 수정
  2. 새 기능 추가 (기존 패턴 따라서)
  3. 테스트 코드 자동 생성
  4. AI 플랫폼 프로젝트에서 실제 작업 발췌
```

## 잘 쓰기 위한 실전 팁

### CLAUDE.md = 프로젝트의 매뉴얼

- AI가 세션 시작할 때 읽는 파일
- 프로젝트 구조, 규칙, 컨벤션을 적어두면 매번 설명 불필요
- **실제 사용 중인 템플릿 예시:**

```draft
- CLAUDE.template.md에서 핵심 섹션 3개만 발췌:
  1. Project Summary — "이렇게 프로젝트를 설명하면 AI가 맥락을 잡는다"
  2. Architecture Rules — "이건 절대 하지 마를 정의"
  3. Commit Rules — "산출물의 품질 기준을 설정"
- 전체 템플릿은 세미나 후 공유 자료로
```

### 맥락 관리

- 컨텍스트 윈도우는 유한하다
- 관련 없는 파일을 열지 않기
- 큰 작업은 나눠서 세션 분리
- **AI의 작업 메모리를 아끼는 것도 "명료성"의 일부**

### 좋은 지시 vs 나쁜 지시

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "이거 고쳐줘" | "이 함수의 반환값이 null일 때 크래시. null 체크 추가해줘" |
| "더 좋게 만들어줘" | "이 루프가 O(n^2). 해시맵으로 O(n)으로 바꿔줘" |
| "테스트 짜줘" | "이 함수의 경계값 케이스 3개에 대한 유닛 테스트 작성해줘" |

- **의도 + 맥락 + 구체적 기대 결과** = 좋은 지시의 공식
- 다시 테제로: **욕구가 명료하면, AI가 만든다**

# Part 7: 나만의 워크플로우를 만들어라 (10분)

## 전통적 워크플로우의 전제를 의심하라

- 브랜치, PR, 코드 리뷰 = **인간 개발자의 역량 한계**를 전제한 협업 구조
  - 한 사람이 전체를 커버할 수 없으니 → 분업 → 합치기 → 검증
- AI 시대: **1인 커버리지가 극적으로 넓어짐**
  - 이전에 팀이 필요했던 규모를 1인이 다룰 수 있음
  - 협업이 불필요해진 게 아니라, **협업의 성격이 바뀜**
  - "네가 짠 코드"의 리뷰 → "네가 AI에게 시킨 코드"의 리뷰

```draft
- 도발적 주장이므로 프레이밍 주의
- "팀이 필요 없다"가 아니라 "개인의 역할이 커졌다"
- 팀 리드급 청중의 반발 예상 → "협업의 질이 올라간다"로 착지
```

## Skill과 Agent: 워크플로우의 빌딩 블록

- **Skill** = 반복되는 작업 패턴을 정의한 것
  - 예: "토론하기", "계획 세우기", "구현하기"
  - AI에게 매번 설명하는 대신, 한 번 정의해두고 재사용
- **Agent** = 특정 전문 영역을 담당하는 하위 AI
  - 예: "API 문서 조사", "테스트 실패 분석"
  - 메인 AI가 필요할 때 호출

```draft
- 실제 skill 파일 하나를 잠깐 보여주는 정도 (10초)
- 내부 구조를 설명하지 않고 "이런 게 가능하다"만
```

## 현실적인 시작 경로

1. **CLAUDE.md부터** — Skill 없이도 즉시 가능
   - 프로젝트 맥락을 파일로 정리하는 것만으로 품질 향상
   - 오늘 세미나 끝나고 바로 할 수 있는 것

2. **반복 패턴 발견 → Skill화**
   - "이 지시를 세 번째 반복하고 있네" → 파일로 추출
   - 처음부터 만들지 않기. 반복이 보일 때.

3. **남의 것을 복사하지 말고 참고만**
   - 맥락이 다르면 워크플로우도 달라야 함
   - 인터넷에서 가져온 Skill을 그대로 쓰면 오히려 방해
   - **나만의 작업 패턴을 자각하는 것**이 시작

```draft
- "특출한 개발자만 가능한 것"이 아님을 강조
- 1→2→3이 자연스러운 성장 경로
- 핵심: meta work process를 최적화하는 감각
```
