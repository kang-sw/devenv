## Chatbot vs Agent: 뭐가 다른가

- Chatbot (복붙 워크플로우)
  - 사람이 맥락을 복사해서 주입
  - AI가 결과를 생성
  - 사람이 결과를 복사해서 붙여넣기
- Agent 워크플로우
  - AI가 필요한 맥락을 **자율적으로 수집**
  - 파일을 읽고, 검색하고, 명령을 실행
  - 사람은 **의도와 검증**에 집중

```
[Chatbot]                    [Agent]
사람 → 맥락 복사 → AI       사람 → 의도 전달 → AI
AI → 결과 생성               AI → 맥락 자율 수집
사람 ← 결과 복사 ← AI            → 실행
                             사람 ← 결과 검증
```

```draft
- 화살표의 굵기/색으로 "사람의 노동"이 어디에 집중되는지 시각화
```

## Token: AI의 연료를 이해하자

- Token = AI가 텍스트를 처리하는 단위
  - 영어: 단어 ≈ 1~2 토큰
  - 한국어: 글자 하나 ≈ 2~3 토큰 (같은 의미에 2~3배 비용)
- Context Window = AI의 작업 메모리
  - 유한한 크기 (현재 최대 100K~200K 토큰)
  - 프로젝트 파일 + 대화 내용 + AI 사고 = 전부 여기에 들어감
  - **한국어로 채우면 같은 메모리에 절반의 정보만 담김**

```draft
- 여기서 "그래서 AI 산출물은 영어로" 논지 전개
- 하지만 저항을 줄이는 프레이밍 필요:
  "영어로 일하라는 게 아니다. AI에게 더 넓은 작업 공간을 주는 것."
- 실전 워크플로우:
  1. AI가 영어로 문서/코드/커밋 작성
  2. 사람이 필요할 때: "이거 한국어로 3줄 요약해줘"
  3. → AI가 즉시 다이제스트 제공
- 시연 가능한 포인트: 영어 기술문서 → 한국어 요약 라이브 데모
```

# Part 3: 개발자의 역할이 바뀌고 있다

## 두 개의 축

```
        구조를 이해하지 못함
              |
              |  (여기는 위험 지대)
              |
   손으로 ----+---- AI에게
   개발       |     맡김
              |
              |  (여기가 목표)
              |
        구조를 이해함
```

- X축: 손으로 개발 ←→ AI에게 위임
- Y축: 이해의 깊이
  - 구조를 이해하지 못함 (위험)
  - 구조를 이해함 (작동 가능)
  - 내용을 이해함 (과거의 이상)

```draft
- 핵심 주장: "내용을 이해하는 건 에이전틱 시대에 불가능해졌다"
  - AI가 생성하는 코드의 양과 속도 > 인간의 리뷰 능력
  - 그런데... 손으로 개발할 때 우리는 얼마나 이해하고 있었나?
  - 프레임워크 내부, 라이브러리 동작, 컴파일러 최적화...
  - 우리는 항상 추상화 위에서 일했다. AI는 레이어를 하나 더 올린 것.
- 위험한 조합: "AI에게 맡김 + 구조를 이해하지 못함" = 블랙박스
- 목표 조합: "AI에게 맡김 + 구조를 이해함" = 아키텍트 + 검증자
- 청중에게 착지: "공부 안 해도 된다는 뜻이 아니다.
  공부의 방향이 바뀐다. 코드 문법 → 시스템 설계."
```
```reply
- 그래서 중요해지는 게 Test의 개념
- 지금 우리가 작업하는 코드에서 어느 부분까지가 "사람 개입 없이 테스트 가능한지"?
- `int add(int, int)` vs `void move_motor_to(float x, float y)` vs `float read_sensor(int index);`의 사례
  - 유닛 테스트는 쉽다. 결합 테스트는 이해해야 한다. 중요한 건 테스트 불가능한 걸 골라내고 모아서 한 번에 검증하는 것
  - 테스트 가능한 부분은 AI가 feedback loop를 돌 수 있음.
```

## 추상화 레이어의 이동

- 과거: 코드 한 줄 이해 → 모듈/패턴 이해
- 현재: 모듈/패턴 이해 → **시스템 의도/제약 이해**
- 개발자의 역할: 코드 작성자 → **아키텍트 + 검증자**

```draft
- 이걸 한 장의 계단 다이어그램으로
- 각 시대의 "개발자가 직접 다루는 레이어" 강조
- 어셈블리 → C → 고급 언어 → 프레임워크 → AI
  매번 "이전 레이어는 이제 몰라도 된다"고 했지만
  "이전 레이어의 구조는 알아야 했다"
```

# Part 4: AI는 당신 편이 아니다

## 인지 강화의 함정

- AI 모델은 사용자의 인지를 **강화**하는 방향으로 작동
  - 사용자가 옳으면 → 더 옳은 방향으로
  - 사용자가 틀리면 → **더 틀린 방향으로** (sycophancy)
- "네, 맞습니다"는 AI의 가장 위험한 대답

```draft
- 구체적 실패 사례 3가지 준비:
  1. AI가 자신만만하게 틀린 코드를 내놓은 사례
  2. AI가 사용자의 잘못된 전제에 동의한 사례
  3. 같은 질문을 다르게 물었더니 완전히 다른 답이 나온 사례
- "의심해라"가 아니라 "이렇게 속는다"를 먼저 보여주기
- 의심은 보여주면 자연스럽게 따라온다
```
```reply
- 틀린 코드를 경험한 사람은 많음
- 잘못된 전제에 인지 강화를 겪은 사람들도 많음.
    - 구체적인 예시 없더라도; 또는 이번 gemini의 자살 사고 사례 같은 걸 소개해도 될듯 
- 같은 질문 두 개의 뉘앙스 차이를 보여주는 게 제일 인상적일 듯.
```

- 다른 세션/모델에게 교차 검증 시키기
- "맞아?"라고 묻지 말고 "여기서 뭐가 틀릴 수 있어?"라고 묻기

```draft
- 실전 팁 위주로 구성
- 청중이 바로 써먹을 수 있는 질문 패턴 예시
```
```reply
- 근데 그 테스트조차 기만적인 사례가 간혹 발견
- 간단한 C언어 예시 코드 보여주면 좋을듯. (더하기 함수에서 항상 3을 반환하게 짜고 값이 3인지 검사하는 등)
```

# Part 6: 나만의 워크플로우를 만들어라

## 전통적 워크플로우의 전제

- 브랜치, PR, 코드 리뷰 = **인간 개발자의 역량 한계**를 전제한 협업 구조
- 한 사람이 전체를 커버할 수 없으니 → 분업 → 합치기 → 검증
- AI 시대: **1인 1프로젝트 커버가 가능해짐**
  - 타인의 워크플로우에 맞추기보다
  - 나의 워크플로우를 120% 최적화하는 게 효과적

```draft
- 이 주장은 도발적. 청중 중 팀 리드급이 있으면 반발 가능.
- "팀 협업이 필요 없다"가 아니라
  "개인의 커버리지가 넓어지면 협업의 성격이 바뀐다"로 착지.
- 코드 리뷰의 의미 변화:
  "네가 짠 코드" → "네가 AI에게 시킨 코드"의 리뷰
```

## Skill과 Agent: 워크플로우의 빌딩 블록

- **Skill** = 반복되는 작업 패턴을 정의한 것
  - 예: "토론하기", "티켓 쓰기", "구현하기"
  - AI에게 매번 설명하는 대신, 한 번 정의해두고 재사용
- **Agent** = 특정 전문 영역을 담당하는 서브 AI
  - 예: "API 문서 조사", "테스트 실패 분석"
  - 메인 AI가 필요할 때 호출

```draft
- 너무 깊이 들어가지 않기. "이런 게 가능하다" 수준.
- 실제 skill 파일 하나를 잠깐 보여주는 정도.
```

## 현실적인 시작점

1. **CLAUDE.md부터** — Skill/Agent 없이도 즉시 가능
   - 프로젝트 맥락을 파일로 정리하는 것만으로 품질 향상
2. **반복 패턴 발견 → 그때 Skill화**
   - "이 지시를 세 번째 반복하고 있네" → Skill로 추출
   - 처음부터 만들지 않기
3. **남의 것을 복사하지 말고 참고만**
   - 맥락이 다르면 워크플로우도 달라야 함
   - 인터넷에서 가져온 Skill을 그대로 쓰면 오히려 방해

```draft
- 이 섹션이 "특출한 개발자만 가능한 것"이 아니라
  "점진적으로 누구나 접근 가능한 것"으로 느껴져야 함
- 1→2→3 순서가 자연스러운 성장 경로
```
