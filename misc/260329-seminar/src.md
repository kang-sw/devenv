# 세미나 소스 문서: "그림이 선명하면, AI가 만든다"

이 문서는 세미나의 소스 오브 트루스이다. 모든 의도, 프레이밍, 맥락, 주의사항이
여기에 담긴다. 실제 슬라이드 워딩은 별도의 `ppt.md`에서 이 문서를 참조하여 증류한다.

## 메타 정보

- **발표자**: 제조업 비전 개발자. AI 플랫폼 1인 개발 경험. Agentic workflow를 진지하게 파고 있음.
- **청중**: 같은 회사 개발자들. GUI(Visual Studio) 중심. AI 경험은 "챗봇 복붙" 수준이 기준선.
  레벨은 다양하나, 기준선을 intermediate 이하로 잡는다.
- **시간**: 120분. Part 3 후, Part 5 중간에 인터랙션 타임 각 5분.
  시간 초과 시 Part 6 데모를 축약.
- **관통 테제**: "그림이 선명하면, AI가 만든다." 모든 파트가 이 테제의 변주.
- **핵심 메타포**: AI는 도구가 아니라 부하 직원이다.
  똑똑하고, 순종적이고, 실행력이 강하고, 잘 넘겨짚는다.

# Part 1: 왜 지금 이 이야기를 하는가 (10분)

## 의도

"AI로 이 정도가 가능하다"를 사례로 보여준 뒤,
비결이 코딩 실력이 아니라 명료성이라는 것으로 전환한다.

## 1-1. 사례: 1인 개발, 1개월, MVP

내용:
- 사내 AI 학습 플랫폼 개발 사례
  - 그래프 노드 기반 웹 플랫폼 (InvokeAI 유사, 단 추론이 아니라 학습 플랫폼)
  - 비동기 스케줄링 중심으로 내부 구현 전면 재구현
  - 바닥부터 1인 개발. 실제 MD(man-day) 투입 10일 내외.
  - 기존 SI 업체에서 특급 개발자 여러 명이 붙어도 될지 모르는 규모
- 단순 코딩 속도에서 AI를 이길 수 없다.

PPT 노트:
- 프로젝트 스크린샷 또는 짧은 데모 영상을 보여준다.
- 숫자만 말한다: 1인, 10일, MVP.

## 1-2. 비결

내용:
- 원하는 걸 구체적으로 말하면 AI가 단숨에 달성한다
- 모호하게 말하면 모호한 구현이 나온다. 버그로 수렴한다.
- **머릿속 그림이 선명하고 명료해야 한다**
- 이 세미나는 그 "명료성"이 무엇인지에 대한 이야기

# Part 2: Chatbot에서 Agent로 (10분)

## 의도

복붙 워크플로우와 Agent 워크플로우의 질적 차이를 보여준다.
Claude Code를 소개하고, 보안 문제를 다루어 이후 내용에 대한 신뢰를 깔아둔다.

## 2-1. 복붙의 한계

내용:
- Chatbot 워크플로우
  - 사람이 맥락을 복사해서 AI에게 전달
  - AI가 결과를 생성
  - 사람이 결과를 복사해서 프로젝트에 붙여넣기
- 문제: 맥락의 병목이 사람
  - 어떤 파일을 줘야 하는지 사람이 판단해야 함
  - 프로젝트가 커지면 사람이 맥락을 다 주는 게 불가능

## 2-2. Agent: 맥락을 AI가 수집한다

내용:
- Agent 워크플로우
  - 사람은 의도를 전달
  - AI가 필요한 맥락을 자율적으로 수집 (파일 읽기, 검색, 명령 실행)
  - 사람은 승인과 검증에 집중
- 비교 다이어그램:

```
[Chatbot]                        [Agent]
사람 → 맥락을 골라서 복사 → AI   사람 → "이거 해줘" → AI
AI → 결과 텍스트 생성             AI → 파일 탐색, 코드 검색 ...
사람 ← 결과를 골라서 붙여넣기          → 수정안 제시
                                 사람 ← 승인 / 거부 / 수정 요청
```

PPT 노트:
- 좌우 비교 레이아웃.

## 2-3. Claude Code: 터미널 위의 Agent

내용:
- AI Agent는 텍스트 기반으로 동작
  - 파일 읽기/쓰기, 검색, 빌드, 테스트 — 전부 터미널 명령
  - GUI에서 마우스로 하던 일을 AI는 텍스트 명령으로 수행
- 터미널이 어려운 게 아니다. AI가 터미널을 대신 쓴다.
  - `claude` 명령 하나로 시작
  - AI가 명령을 제안 → 사람이 승인/거부 → 실행

PPT 노트:
- "터미널을 배워라"가 아님을 명확히 한다.
- 다만 ls, cd, git 정도는 알아야 승인/거부 판단이 가능하다.
- 가능하면 여기서 claude 실행 → 간단한 요청 → 승인 과정을 짧게 시연.

## 2-4. 보안

내용:
- AI Agent는 로컬 파일을 읽고 명령을 실행한다.
  민감 데이터를 다루는 환경에서 이건 진짜 문제.

- 보안 대상을 구분하자:
  - 범용 로직 (CRUD, 유틸리티, 인프라 코드) — 보호 우선순위가 높지 않음
  - 도메인 특화 알고리즘 (비전 검사 로직, 고객별 특수 처리) — 보호 대상
  - 고객사 민감 데이터 (도면, 사양, 라인 정보) — 절대 보호 대상
  - 전부 잠그면 AI를 쓸 수 있는 영역이 불필요하게 좁아진다.
    대상을 구분하면 쓸 수 있는 영역이 생각보다 넓다.

- 이건 "쓸지 말지"가 아니라 "언제, 어떻게"의 문제:
  - 생산성 혁신이 일어나고 있다는 건 기정사실
  - 전사적 도입에 앞서, 각 개인이 준비되어 있어야 한다

- 지금 할 수 있는 것 — 개인 준비:
  - 개인 프로젝트, 비업무 코드로 워크플로우를 익혀두기
  - 조직이 도입을 결정했을 때, 준비된 사람과 그렇지 않은 사람의 차이는 크다

- 조직 차원의 경로:
  - 엔터프라이즈 계약 (데이터 학습 금지 조항, SOC2 인증)
  - 온프레미스 배포, 로컬 모델 (성능은 떨어지지만 개선 중)
  - 옵션이 있다는 걸 아는 것만으로도 사내 논의가 달라진다

PPT 노트:
- 톤: "왜 안 쓰냐"가 아니라 "준비해두면 기회가 왔을 때 잡는다."
- 보수적 시선을 존중하되, "아무것도 안 한다"로 끝나지 않게.
- 기존 보안 상태를 지적하지 않는다.

# Part 3: AI의 연료를 이해하자 (10분)

## 의도

AI의 물리적 제약을 이해시킨다. 토큰, 컨텍스트 윈도우, 비용.
"영어 워크플로우"의 근거를 역할 분리의 관점에서 제시한다.

## 3-1. Token이란

내용:
- AI가 텍스트를 처리하는 최소 단위
- 영어: 단어 ≈ 1~2 토큰
- 한국어: 음절 하나 ≈ 2~3 토큰
- 같은 의미를 전달하는 데 한국어는 2~3배 토큰을 소모한다

PPT 노트:
- 같은 문장을 한국어/영어로 놓고 토큰 수를 비교한다.
- 조사, 부사 등이 전부 개별 토큰으로 소모되는 걸 시각적으로 보여준다.
- 토크나이저 도구로 실시간 시연하면 가장 효과적.
- 극적인 차이가 나는 문장을 미리 선정해 둘 것.

## 3-2. Context Window: AI의 작업 메모리

내용:
- AI가 한 번에 볼 수 있는 전체 텍스트 크기
  - 프로젝트 파일 + 대화 내용 + AI의 사고 = 전부 여기에 들어감
- 한국어로 채우면 같은 메모리에 절반의 정보만 담긴다
- 비유: AI의 작업 메모리는 책상이다. 한국어로 채우면 같은 책상에 절반의 자료만 올릴 수 있다.

크기보다 밀도:
- 1M 토큰 모델도 있고 (Gemini 등), 이미 쓰고 있는 사람도 많을 것
- 하지만 책상을 넓힌다고 일을 잘하는 게 아니다
  - 넓은 책상에 자료를 늘어놓을수록 정작 중요한 문서가 눈에 띄지 않는다
  - 컨텍스트가 길어지면 정작 중요한 정보에 대한 AI의 집중도가 떨어짐
- 현재 시점에서는 핵심 맥락을 200K 이내로 정제하는 게 더 나은 결과를 낸다
- **정보의 양이 아니라 정보의 밀도가 품질을 결정한다**

PPT 노트:
- 특정 모델을 깎아내리지 않는다.
- "lost in the middle" 같은 기술 용어는 쓰지 않는다. 책상 비유로 충분.
- "현재 시점에서는"이라는 한정어를 붙인다. 빠르게 개선 중인 영역.

## 3-3. AI 산출물은 영어로

내용:
- 이 세미나의 테제: 사람은 구조와 의도, AI는 디테일.
- 이 역할 분리에서 언어 분리가 따라온다:
  - AI가 다루는 것 (코드, 문서, 커밋, 기술 산출물) = 디테일 = 영어
  - 사람이 다루는 것 (논의, 가이던스, 의사결정) = 구조/의도 = 한국어
- 영어로 일하라는 게 아니다. 역할이 나뉘면 언어도 나뉜다.
- 토큰 효율은 부가 이익:
  AI가 영어로 작업하면 같은 컨텍스트에 2~3배 많은 정보를 담을 수 있다

- 실전 워크플로우:
  1. 사람은 한국어로 의도/방향을 전달
  2. AI는 영어로 모든 산출물을 작성
  3. 사람이 디테일을 확인해야 할 때: "한국어로 핵심만 요약해줘"
  4. AI가 즉시 다이제스트 제공
- 사람은 모국어로 생각하고, AI는 효율적인 언어로 일하고,
  필요할 때만 번역 레이어가 끼어든다.

PPT 노트:
- 시연: 영어 기술문서 → "한국어로 핵심만 요약해줘" → 즉시 결과.
- 영어 → 한국어 다이제스트 방향으로 보여줘야 저항이 적다.

## 3-4. 비용의 현실

내용:
- 시작: 월 $20 (Claude Pro) — 학습용, 일상적 보조 도구
- 도달점: 월 $200 (Max) — 본업의 생산성을 바꾸는 수준. 발표자는 여기.
  - 실제 API 환산 시 월 $2,000어치 사용 중
  - 이 가격 구조가 언제까지 유지될지는 미지수
- 계산:
  - $200/월 = 하루 $7 ≈ 커피 두 잔
  - 그 대가로 1인 1개월 MVP 수준의 산출물
  - SI 업체 인건비와 비교하면?
- 이 도구가 $200의 가치가 있는지는, 써봐야 안다. 시작은 $20.
- 이 비용 구조는 지속되지 않을 수 있다. 지금이 기회.

PPT 노트:
- $20으로는 실질적 Agent 워크플로우가 어렵다는 점을 말하되,
  학습과 감 잡기에는 충분하다고 안내한다.
- 불확실성도 인정한다. 과장하지 않는다.

# Part 4: 개발자의 역할이 바뀌고 있다 (20분)

## 의도

세미나의 피벗 포인트. 앞은 "이게 뭔지", 뒤는 "어떻게 쓸지".
여기서 "왜 이렇게까지 해야 하는지"의 동기를 제공한다.

## 4-1. 두 개의 축

내용:

```
          이해 없음
              |
              |  위험 지대:
              |  AI에 의존하지만
              |  뭘 하는지 모름
              |
   손으로 ----+---- AI에게
   개발       |     위임
              |
              |  목표 지대:
              |  AI에게 맡기되
              |  구조를 꿰뚫음
              |
          구조 이해
```

- X축: 직접 코딩 ←→ AI에게 위임
- Y축: 이해의 깊이
- 위험한 조합: AI에게 맡김 + 구조를 모름 = 블랙박스
- 목표 조합: AI에게 맡김 + 구조를 이해함 = 아키텍트 + 검증자

## 4-2. "내용을 이해한다"는 환상

내용:
- 에이전틱 시대에 코드 한 줄 한 줄을 이해하는 건 불가능해졌다
  - AI가 생성하는 양과 속도가 인간의 리뷰 능력을 넘는다
- 그런데, 손으로 개발할 때 우리는 얼마나 이해하고 있었나?
  - 프레임워크 내부? 라이브러리 동작? 컴파일러 최적화?
  - 우리는 항상 추상화 위에서 일했다
  - AI는 그 레이어를 하나 더 올린 것뿐

## 4-3. 추상화 레이어의 역사 — 그리고 단절

내용:

```
어셈블리 → C → 고급 언어 → 프레임워크 → ???
```

- 매번 "이전 레이어는 이제 직접 안 다뤄도 된다"
- 하지만 매번 "이전 레이어의 구조는 알아야 했다"
- 여기까지는 공통점. 그런데 결정적 차이가 있다:

비결정성의 등장:
- 어셈블리부터 프레임워크까지: 같은 코드 → 같은 결과. 결정적.
- AI: 같은 프롬프트 → 다른 결과. 비결정적.
- 이건 도구의 영역이 아니다.
- **결국 사람에게 일을 맡기는 것과 같다.**

프레임 전환:
- 우리가 얻은 건 새로운 도구가 아니라,
  똑똑하고, 순종적이고, 실행력이 강하고, 잘 넘겨짚는 부하 직원.
- 부하 직원이 "네 맞습니다" 하면? → 의심해야 한다
- 부하 직원에게 지시할 때는? → 명확해야 한다
- 부하 직원의 산출물은? → 리뷰해야 한다

PPT 노트:
- 계단 다이어그램에서 마지막 칸에 "???"을 두고, "이건 도구가 아닙니다"로 전환.
- 착지: 공부 안 해도 된다는 뜻이 아니다. 공부의 방향이 바뀐다.
  코드 문법 → 시스템 설계. 구현 기술 → 검증 기술.

# Part 5: AI는 당신 편이 아니다 (20분)

## 의도

"부하 직원" 메타포를 이어받아, 만능론을 교정한다.
위험과 대응을 교차 배치하여 "문제 → 즉시 해결책" 리듬을 유지한다.

## 5-1. 인지 강화의 함정 → 대응: 질문을 바꿔라

위험:
- AI는 사용자의 인지를 강화하는 방향으로 작동한다
  - 사용자가 옳으면 → 더 옳은 방향으로 가속
  - 사용자가 틀리면 → 더 틀린 방향으로 가속 (sycophancy)
- "네, 맞습니다"는 AI의 가장 위험한 대답

시연/사례:
- 라이브 시연: 같은 질문을 두 가지 뉘앙스로 던져서 다른 답을 받는다.
- 자신만만하게 틀린 코드를 내놓은 본인 경험.

즉시 대응:
- 질문을 바꿔라
  - X: "이거 맞아?" → AI: "네, 맞습니다"
  - O: "여기서 뭐가 틀릴 수 있어?" → AI가 약점을 찾는다
- 교차 검증
  - 다른 세션이나 다른 모델에게 같은 산출물을 리뷰시킨다
  - 한 AI의 확신은 증거가 아니다

## 5-2. 테스트 가능성의 스펙트럼

내용:

```c
// Level 1: AI 혼자 검증 가능
int add(int a, int b) { return a + b; }
// → 유닛 테스트 작성 + 실행 + 결과 확인까지 AI가 혼자 가능

// Level 2: 구조 이해가 필요
void move_motor_to(float x, float y);
// → mock/시뮬레이션이 필요. 뭘 mock할지는 사람이 설계

// Level 3: 사람만 검증 가능
float read_sensor(int index);
// → 하드웨어 의존. 실제 동작은 사람이 눈으로 확인
```

- Level 1: AI가 코드를 쓰고, 테스트를 쓰고, 실행하고, 실패하면 고치고, 다시 실행한다.
  사람 개입 없이 목적을 달성할 때까지 반복한다. 이것이 feedback loop.
  부하 직원 비유로: 결과를 스스로 확인하고 고칠 수 있는 일.
  이 사이클을 AI가 혼자 돌 수 있으면, 그 영역은 맡겨도 된다.

- Level 2: mock/시뮬레이션 설계가 개발자의 역할.
  모터가 실제로 움직이지 않아도, 모터에 보내는 명령이 올바른지 검사하는 것을 설계한다.
  부하 직원 비유로: 결과를 상사에게 가져와야만 맞는지 알 수 있는 일.

- Level 3: 하드웨어가 물리적으로 연결된 상태에서만 확인 가능. 사람이 직접.

- 경계는 고정이 아니다:
  Level 3을 mock/시뮬레이션으로 Level 2로 끌어내리는 것,
  Level 2를 자동화 테스트로 Level 1로 끌어내리는 것 —
  **이 경계를 밀어내는 것이 지금 시대 개발자의 일이다.**

PPT 노트:
- feedback loop를 "AI가 혼자서 시행착오를 반복하는 것"으로 풀어 설명한다.
- 청중의 실무(모터, 센서, 비전)에 직접 매핑되는 예시를 쓴다.

## 인터랙션 B (5분)

## 의도

기만적 안전망을 보여주기 전에, 청중이 직접 "테스트를 믿었다가 당하는" 경험을 한다.

## 진행

화면에 코드를 띄우고:
> "이 코드와 테스트가 있습니다. 버그를 찾아보세요. 10초."

```c
int add(int a, int b) {
    return 3;
}

assert(add(1, 2) == 3);  // PASS!
```

10초 후 답을 공개. 이어서 5-3으로 연결.

PPT 노트:
- 짧지만 각성 효과가 크다.
- "테스트가 통과했는데 틀렸다"를 머리가 아니라 몸으로 경험한다.

## 5-3. 기만적인 안전망 → 대응: 테스트의 의도를 사람이 잡는다

위험:
- 테스트가 있다 ≠ 안전하다
- AI는 "테스트를 통과시키는 것"에 최적화할 수 있다
- 사람이 봐야 하는 건 코드가 아니라 테스트의 의도
  - "이 테스트가 진짜 의미 있는 걸 검증하고 있나?"

즉시 대응:
- 테스트 먼저, 구현 나중 (TDD)
  - 테스트의 의도를 사람이 먼저 정의
  - 구현은 AI에게 맡기기
  - 통과하면 다음으로. 실패하면 AI에게 수정 지시.
- 내 말로 다시 설명하기
  - AI의 설명을 듣고 "아 네" 하면 위험
  - 내 말로 다시 이야기할 수 있어야 이해한 것
  - "그러니까 이건 이런 거지?" — 이 한 마디가 검증
  - 이걸 꾸준히 하면 나와 AI 사이에 공유된 인지 지도가 만들어진다
  - 그 지도가 있으면 짧은 지시만으로 AI가 정확히 움직인다

PPT 노트:
- "내 말로 다시 설명하기"는 Part 3의 다이제스트 개념과 연결된다.
  Part 3에서는 "영어→한국어 요약"이라는 물리적 행위로 소개했고,
  여기서는 이해의 검증 도구로 회수.

# Part 6: 실전 — 이렇게 일합니다 (20분)

## 의도

이론에서 실전으로 전환한다. 실제 작업 과정을 보여주고 실전 팁을 전달한다.

## 6-1. [녹화 데모] 실제 프로젝트 워크스루

방식: 녹화 + 해설. 배속 편집으로 AI 대기 시간 제거.

시나리오: AI 플랫폼 프로젝트에서 실제 작업 발췌.

구성:
1. 논의 시작 장면 — "이런 기능이 필요하다"에서 대화가 시작되는 모습
2. AI가 생성한 멘탈 모델 문서 — AI와 사람이 공유하는 프로젝트 구조 지도
3. 구현 과정 — 핵심 부분만 배속. AI가 맥락을 수집하는 과정을 보여준다.
4. 완성된 기능

PPT 노트:
- 코딩 시연이 아니라 협업 과정 시연. 청중이 보는 건 코드가 아니라 대화 방식.

## 6-2. 잘 쓰기 위한 실전 팁

### CLAUDE.md = 프로젝트의 매뉴얼

내용:
- AI가 세션 시작할 때 읽는 파일
- 프로젝트 구조, 규칙, 컨벤션을 적어두면 매번 설명 불필요

PPT 노트:
- CLAUDE.template.md에서 핵심 섹션 3개만 발췌하여 보여준다:
  1. Project Summary
  2. Architecture Rules
  3. Commit Rules
- 전체 템플릿은 세미나 후 공유.

### 맥락 관리

내용:
- 컨텍스트 윈도우는 유한하다
- 관련 없는 파일을 열지 않기
- 큰 작업은 나눠서 세션 분리
- 기존 코드베이스를 통째로 AI에게 던지면 할당량이 금방 소진된다.
  프로젝트 문서(CLAUDE.md, 멘탈 모델)로 맥락을 정제해 두고,
  AI가 필요한 부분만 볼 수 있게 안내하는 것이 핵심.
- AI의 작업 메모리를 아끼는 것도 명료성의 일부

### 좋은 지시 vs 나쁜 지시

| 나쁜 예 | 좋은 예 |
|---------|---------|
| "이거 고쳐줘" | "이 함수의 반환값이 null일 때 크래시. null 체크 추가해줘" |
| "더 좋게 만들어줘" | "이 루프가 O(n²). 해시맵으로 O(n)으로 바꿔줘" |
| "테스트 짜줘" | "이 함수의 경계값 케이스 3개에 대한 유닛 테스트 작성해줘" |

- 의도 + 맥락 + 구체적 기대 결과 = 좋은 지시의 공식

# Part 7: 나만의 워크플로우를 만들어라 (10분)

## 의도

지금 당장 할 수 있는 것부터 장기적 성장 방향까지의 경로를 제시한다.

## 7-1. 현실적인 시작 경로

내용:
1. CLAUDE.md부터 — Skill 없이도 즉시 가능
   - 프로젝트 맥락을 파일로 정리하는 것만으로 품질 향상
   - 오늘 세미나 끝나고 바로 할 수 있는 것

2. 반복 패턴 발견 → Skill화
   - "이 지시를 세 번째 반복하고 있네" → 파일로 추출
   - 처음부터 만들지 않는다. 반복이 보일 때.

3. 남의 것을 복사하지 말고 참고만
   - 맥락이 다르면 워크플로우도 달라야 한다
   - 인터넷에서 가져온 Skill을 그대로 쓰면 오히려 방해
   - 나만의 작업 패턴을 자각하는 것이 시작

PPT 노트:
- 1→2→3이 자연스러운 성장 경로.
- 1은 오늘, 2는 한 달, 3은 석 달 정도.

## 7-2. 전통적 워크플로우의 전제를 의심하라

내용:
- 브랜치, PR, 코드 리뷰 = 인간 개발자의 역량 한계를 전제한 협업 구조
  - 한 사람이 전체를 커버할 수 없으니 → 분업 → 합치기 → 검증
- AI 시대: 1인 커버리지가 넓어졌다
  - 이전에 팀이 필요했던 규모를 1인이 다룰 수 있음
  - 협업이 불필요해진 게 아니라, 협업의 성격이 바뀜
  - "네가 짠 코드"의 리뷰 → "네가 AI에게 시킨 코드"의 리뷰

PPT 노트:
- "팀이 필요 없다"가 아니라 "개인의 역할이 커졌다"로 말한다.

## 7-3. 성장 방향: Skill, Agent, 그리고 메타

내용:
- 더 나아가면 반복 패턴을 Skill로, 전문 영역을 Agent로 정의할 수 있다
  - Skill = "토론하기", "계획 세우기", "구현하기" 같은 작업 패턴
  - Agent = "API 문서 조사", "테스트 실패 분석" 같은 전문 하위 AI
- 요리를 잘하려면 레시피를 따라가면 된다. (= 일)
  어떤 사람은 레시피를 만든다. (= 메타 일)
  Skill을 만드는 건 자기만의 레시피를 쓰는 것이다.

PPT 노트:
- 실제 skill 파일 하나를 잠깐 보여준다. 내부 구조는 설명하지 않는다.
- 이 섹션은 "이런 세계가 있다"는 방향 제시. 깊이 들어가지 않는다.

슬라이드에 넣지 않는다. 예상 질문과 답변 프레임만 준비.

## "우리는 이제 어떻게 되나요?"

예상 맥락:
- 현재 SW 수요를 충족하는 건 AI를 다루는 개발자 2~3명이면 끝
- 40명 넘는 나머지 인력은?
- 일부 개발자가 오퍼레이터로 격하될 가능성

답변 프레임:
1. 인정: "현재의 SW 수요만 본다면 그 우려는 타당합니다."
2. 전환: "'현재 수요를 누가 충족하나'가 아니라 '수요 자체를 얼마나 키울 수 있나'가 질문이어야 합니다."
3. 특수성: "우리 업종은 SW로 풀 수 있지만 아직 풀지 않은 문제가 산더미입니다."
4. 현실: "이 전환에 올라타는 사람과 못 타는 사람의 격차는 벌어집니다.
   의자 뺏기 싸움이 된 상황에서, 준비하는 수밖에 없습니다."

# 부록: 첫 체험 가이드 (PPT 배포 시 부록으로 첨부)

세미나 후 스스로 따라할 수 있는 절차. 그대로 복붙해도 동작하는 수준으로 작성.

## 사전 준비

1. Claude Code 설치
   - Windows: `winget install Anthropic.ClaudeCode`
   - Mac: `brew install claude-code`
2. Python 설치 (3.10 이상)
   - Windows: `winget install Python.Python.3.12`
   - Mac: `brew install python@3.12`
3. Claude 계정 + Pro 구독 ($20/월)

## 예시 프로젝트: 측정 데이터 관리 API

빈 폴더에서 시작하여, 측정 데이터를 등록/조회/삭제할 수 있는
웹 API 서버를 만든다. 브라우저에서 결과를 확인할 수 있다.

> 이 예시가 취향에 맞지 않으면 자유롭게 바꿔도 된다.
> 핵심은 "빈 폴더에서 시작"하고 "아래 절차를 따르는 것"이다.

## 절차

### Step 0. 폴더 만들기

```
mkdir my-first-project
cd my-first-project
claude
```

### Step 1. 물어봐라 — "이런 걸 만들고 싶다"

AI에게 의도를 전달하고, 구조를 잡아달라고 한다.

> 예시 프롬프트:
> "측정 데이터를 등록하고 조회하고 삭제할 수 있는 간단한 웹 API를
> Python으로 만들고 싶어. 어떤 구조로 만들면 좋을지 제안해줘."

- AI가 기술 스택(Flask/FastAPI 등)과 구조를 제안할 것이다.
- 모르는 용어가 나오면 바로 물어본다: "Flask가 뭐야?"
- 여기서 코드를 짜지 않는다. 구조만 합의한다.

### Step 2. 계획해라 — "뭐부터 만들까"

> "그러면 구현 순서를 잡아줘. 뭐부터 만들면 좋을까?"

- AI가 단계별 계획을 줄 것이다.
- 이해가 안 되는 단계가 있으면 "3번이 무슨 뜻이야?" 하고 물어본다.
- 계획이 납득되면 다음으로.

### Step 3. 시켜라 — "1번부터 구현해줘"

> "1번부터 구현해줘. 테스트도 같이 작성해줘."

- AI가 파일을 만들고, 코드를 쓰고, 테스트를 실행한다.
- 승인/거부 프롬프트가 뜨면 내용을 읽고 판단한다.
  잘 모르겠으면 승인해도 된다 — 빈 폴더에서 시작했으므로 망가질 게 없다.
- 한 단계가 끝나면 "다음 단계 진행해줘"로 이어간다.

### Step 4. 정리해라 — 세션이 길어지면

> `/compact`

- 대화가 길어지면 AI의 작업 메모리가 부족해진다.
- `/compact`를 입력하면 AI가 대화를 요약하고 메모리를 확보한다.
- 체감상 대화가 20회 이상 오갔으면 한 번 해준다.

### Step 5. 확인해라 — 결과를 눈으로 본다

> "서버를 실행해줘"

- AI가 서버를 띄워줄 것이다.
- 브라우저에서 주소를 열어 결과를 확인한다.
- 고치고 싶은 게 있으면 한국어로 말한다:
  "목록이 최신순으로 정렬됐으면 좋겠어" 등.

- **기존 업무 코드에 바로 쓰지 말 것.** 빈 폴더에서 충분히 익힌 뒤에.
